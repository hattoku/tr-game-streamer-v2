## 1. 開発環境セットアップ

### 必須ツール

- Node.js 20.x 以上
- npm
- Firebase プロジェクト（認証情報は `lib/firebase.ts` に直接記述）

### 環境の切り替え（本番環境 / 検証環境）

本プロジェクトでは、用途に応じて本番環境と検証環境（ステージング環境）を切り替えて開発・ビルドを行います。切り替えは npm スクリプトを使用し、内部で環境変数 `NEXT_PUBLIC_APP_ENV` によって制御・切り替えされます。

#### 1. 本番環境（Production）向け
標準のコマンドを使用します。`NEXT_PUBLIC_APP_ENV` は未指定（またはデフォルト扱い）となります。

- **開発サーバー起動**:
  ```bash
  npm run dev
  ```
- **ビルド**:
  ```bash
  npm run build
  ```

#### 2. 検証環境（Staging）向け
検証用の設定で起動・ビルドする場合は `:stg` サフィックスがついたコマンドを使用します。内部で `NEXT_PUBLIC_APP_ENV=stg` が設定されます。

- **開発サーバー起動（検証環境）**:
  ```bash
  npm run dev:stg
  ```
- **ビルド（検証環境）**:
  ```bash
  npm run build:stg
  ```

> **実装時の注意**: アプリケーションコード内で環境による処理（APIエンドポイントや定数の切り替えなど）の分岐が必要な場合は、`process.env.NEXT_PUBLIC_APP_ENV === 'stg'` のように判定を行ってください。

## 2. ドキュメント駆動開発

### 基本原則

**仕様書が主、コードが副。**

- 実装の根拠は必ず仕様書に存在する
- 仕様書に書かれていないことは実装しない
- コードを書く前に、対応する仕様書を読む

### 仕様書の種類と役割

仕様書はすべて `document/` 配下に格納されている。

```
document/
├── plan/           # 企画書・調査レポート（方針・背景）
├── specification/
│   ├── common/     # 共通仕様（技術スタック・権限表・UIコンポーネント等）
│   ├── page/       # ページ別仕様書（画面・ロジック・導線）
│   └── management/ # 管理画面仕様書
└── data/           # マスタデータ定義（ジャンル・タグ等）
```

実装対象（ページ・URL・機能）とどの仕様書が対応するかは、以下を参照。

> 📄 `document/specification/仕様書トレーサビリティ・マトリクス`  
> 📄 `document/specification/page/ページ一覧仕様書`

### 仕様変更の判断基準

以下のいずれかが変わったとき、仕様変更が発生したと判断する。

| 変更の種類 | 例 |
|-----------|---|
| ユーザーに見える挙動 | ボタンの動作・バリデーションルール・エラーメッセージ |
| 画面構成・表示内容 | 表示項目の追加・削除・並び順の変更 |
| ビジネスロジック | 権限ルール・審査フロー・スコア計算式 |

コードのリファクタリング・パフォーマンス改善・変数名の変更など、実装の内部詳細は仕様書に記載しない。仕様書は要件を伝えるドキュメントであり、実装の細部まで追いかけると可読性が損なわれる。

## 3. 開発フロー

### 通常の実装フロー（仕様書 → コード）

```
1. 対応する仕様書を読む（上記マッピング表を参照）
      ↓
2. 仕様書に基づいてコードを書く
      ↓
3. 実装完了
```

### 仕様変更が発生したときのフロー

コードを書く中で、ユーザーに見える挙動・画面・ロジックを仕様書と異なる形で実装する必要が生じた場合、**コードと仕様書を同時に更新する**。

```
仕様書と異なる実装が必要になった
      ↓
【必須】対応する仕様書を更新する
      ↓
コードを書く
      ↓
関連する仕様書が他にもないか確認する（下記「影響範囲の確認」参照）
      ↓
必要であれば関連仕様書も更新する
```

> **NG パターン**: コードだけ変えて仕様書を更新しない。仕様書とコードの乖離は、次の実装者（未来の自分を含む）を混乱させる。

### 影響範囲の確認

一つの変更が複数の仕様書に影響することがある。更新前に以下を確認する。

| 変更した仕様書 | 連鎖して確認すべき仕様書の例 |
|-------------|----------------------|
| ページ仕様書（画面・ロジック） | 共通UIコンポーネント仕様書 / 機能別権限表仕様書 |
| 権限ルール | 機能別権限表仕様書 / 各ページ仕様書 / 管理画面仕様書 |
| 審査フロー | 審査ワークフロー仕様書 / ユーザー管理仕様書 / 通知まわりの仕様書 |
| スコアリングロジック | 信頼度スコアリングシステム仕様書 / レビュー投稿機能仕様書 |
| データ取得バッチ | YouTube定期取得仕様書 / 技術スタック仕様書 |

## 4. コード規約

### ファイル冒頭のコメント

各ファイルの冒頭には日本語のコメントで、そのファイルが何をするものかを記述する。

```typescript
/**
 * 信頼度スコアを計算するユーティリティ
 * 計算式: W = (D + H + G) × C
 * 詳細は `document/specification/common/信頼度スコアリングシステム仕様書` を参照
 */
```

対応する仕様書がある場合は、仕様書へのパスも記載する。

### 仕様書との対応を明示する

仕様書に定義された重要なロジック（権限判定・スコア計算・審査フロー等）を実装するコードには、参照元の仕様書と章番号をコメントで記載する。

```typescript
// 審査者アサインルール: 管理 審査ワークフロー仕様書 第4章
if (!gameTitle) {
  // ゲームタイトルが紐づいていない場合はSTEP 1をスキップしてSTEP 2へ
  return assignToStep2();
}
```

## 5. 秘密情報の管理

### 現在の方針（開発フェーズ）

開発・共有の利便性のために、以下の秘密情報をソースコードに直接記述しリポジトリに含めている。

| ファイル | 内容 |
|---------|------|
| `lib/constants.ts` | YouTube API キー |
| `lib/firebase.ts` | Firebase 認証情報 |

新たに秘密情報を追加する場合も、同様の構成ファイルに追加する。

### 本番環境での管理

本番環境ではすべての秘密情報を **GCP Secret Manager** で管理する。Cloud Functions からは Secret Manager 経由でアクセスする。詳細は `document/specification/common/技術スタック仕様書` を参照。
