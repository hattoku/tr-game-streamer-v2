## 1. 目的

AI運営者がYouTubeを巡回して再生リスト候補を収集する際に、「収集してよい再生リスト」と「除外すべき再生リスト」をAIが判断するための基準を定義する。

この基準はGemini APIへのプロンプト設計（AI運営者_コンテンツ収集機能仕様書）の入力となる。定性的な「良さ」の議論ではなく、AIが判定可能な観点に落とし込むことを重視する。

---

## 2. ゲムコメにおける「優良な再生リスト」の定義

ゲムコメは「ゲーム実況の再生リスト視聴記録＆新着通知サービス」であり、ユーザーが「追いかけたい」と思える実況シリーズを提供することが目的である。この目的に照らして、収集対象とする再生リストを以下のように定義する。

**対象とする再生リスト**:「一人または複数の実況者が、1つのゲームタイトルをプレイする様子を複数回に分けて動画として投稿し、再生リストとしてまとめたもの」

---

## 3. 収集対象の判定基準

以下の基準を**すべて満たす**再生リストを収集対象とする。

### 3.1 必須条件（ハードフィルタ）

AIの判定に先立ち、APIから取得したデータで機械的に確認する条件。

| 条件 | 基準値 | 根拠 |
|------|:------:|------|
| 動画数 | **3本以上** | 1〜2本は「シリーズ」として成立しない |
| 日本語コンテンツ | `relevanceLanguage=ja` かつ再生リストタイトルが日本語を含む | ゲムコメは日本語統一プラットフォーム |
| 再生リストが公開状態 | APIで取得できること | 非公開・削除済みは除外 |

### 3.2 推奨条件（AIによる判定）

必須条件を通過した候補に対し、Gemini APIが以下の観点で判定する。

| 観点 | 判定内容 | 収集推奨の目安 |
|------|---------|-------------|
| **実況シリーズらしさ** | タイトルおよび動画タイトル群から、ゲーム実況・プレイ動画のシリーズであることが読み取れるか | 「第○話」「Part○」「○回目」「クリア」「実況」「プレイ」等の語句を含む |
| **更新継続性** | 動画の投稿間隔が継続的かどうか | 直近の投稿から90日以内、または平均投稿間隔の3倍以内の経過期間 |
| **スパム・荒らし** | 再生リストタイトル・チャンネル名に明らかな不正・無関係なコンテンツが含まれないか | 無関係なキーワードの羅列・連続スパム投稿等が見られない |

## 4. 除外対象の判定基準

以下のいずれかに該当する再生リストは収集候補から除外する。

### 4.1 ハードフィルタで除外（機械的判定）

| 除外条件 | 理由 |
|---------|------|
| 動画数が2本以下 | シリーズとして成立しない |
| 既にゲムコメに登録済みの再生リストID | 重複登録の防止 |
| APIで取得できない（非公開・削除済み） | 視聴不可コンテンツ |

### 4.2 AIが除外フラグを立てる条件

| フラグ名 | 判定基準 |
|---------|---------|
| `not_gameplay` | ゲーム実況・プレイ動画ではないと判断される（攻略解説・レビュー動画・ゲーム無関係コンテンツ等） |
| `update_stopped` | 更新が長期間停止していると判断される（目安：最終投稿から90日超過、かつ平均投稿間隔の3倍以上） |
| `completed` | シリーズが完結済みと判断される（動画タイトルに「最終回」「エンディング」「クリア後」等を含む） |
| `spam` | スパム・荒らし・不正コンテンツの疑いがある |
| `non_japanese` | 日本語コンテンツでないと判断される |

> **フラグがあっても収集候補に含める**: フラグはオーナーが承認判断する際の参考情報として提示する。`completed`（完結済み）や `update_stopped`（更新停止）は、ゲムコメが完結した名作シリーズのアーカイブとしても価値を持つため、自動除外ではなくオーナー判断に委ねる。`spam` のみ自動除外対象とする。

## 5. 検索キーワード設計

AI運営者が `search.list` で使用する検索クエリの設計方針。

### 5.1 基本構成

```
{ゲームタイトル名} 実況
{ゲームタイトル名} プレイ動画
{ゲームジャンル名} 実況 プレイ
```

### 5.2 ジャンルごとのキーワード例

| ジャンル | 検索キーワード例 |
|---------|---------------|
| RPG | `RPG 実況 プレイ`、`JRPG 実況`、`ロールプレイング 実況` |
| シミュレーション | `シミュレーション 実況`、`ストラテジー 実況`、`経営ゲーム 実況` |
| 格闘アクション | `アクション 実況`、`格闘ゲーム 実況`、`アクションRPG 実況` |
| アドベンチャー | `アドベンチャー 実況`、`ホラー 実況`、`脱出ゲーム 実況` |
| シューティング | `シューティング 実況`、`FPS 実況`、`STG 実況` |
| スポーツレース | `スポーツゲーム 実況`、`レースゲーム 実況`、`サッカーゲーム 実況` |
| テーブルパズル | `パズルゲーム 実況`、`ボードゲーム 実況`、`テーブルゲーム 実況` |
| 教育学習 | `知育ゲーム 実況`、`教育ゲーム プレイ` |
| その他 | `インディーゲーム 実況`、`同人ゲーム 実況`、`ゲーム 実況` |

### 5.3 キーワードの管理方針

- 初期キーワードは上記を出発点として運用開始する。
- 収集精度（有効候補の比率）をモニタリングしながら、オーナーがキーワードを追加・削除・変更できる管理画面を設ける（AI運営者_コンテンツ収集機能仕様書にて定義）。
- Gemini APIを使ったキーワードの自動生成・改善については将来検討とする。

## 6. AIによる判定への落とし込み

### 6.1 判定に使用するデータ

Gemini APIに渡すデータの一覧（取得コストとのバランスを考慮して厳選する）。

| データ | 取得元 | 判定での用途 |
|-------|--------|------------|
| 再生リストタイトル | `playlists.list` | 実況らしさ・スパム判定 |
| チャンネル名 | `playlists.list` | スパム判定 |
| 動画数 | `playlists.list` | ハードフィルタ（3本以上）の確認 |
| 動画タイトル一覧（最大30件） | `playlistItems.list` | 実況らしさ・完結判定 |
| 各動画の公開日時（最大30件） | `playlistItems.list` | 更新継続性の判定 |
| 最終動画公開日時 | `playlistItems.list` | 更新停止判定 |

### 6.2 判定結果の出力形式

AI運営者_審査機能仕様書の判定出力形式と同様に、JSON形式で出力する。

```json
{
  "result": "collect" | "skip",
  "flags": ["flag_key", ...],
  "comment": "判定根拠の説明（日本語・200文字以内）"
}
```

| フィールド | 内容 |
|-----------|------|
| `result` | `"collect"`：収集候補に含める / `"skip"`：除外する（スパム判定時のみ） |
| `flags` | 該当するフラグキーの配列（第4章の `not_gameplay` / `update_stopped` / `completed` / `spam` / `non_japanese`） |
| `comment` | 判定根拠の簡潔な説明 |

## 7. 未解決の検討事項

本検討書の策定後も継続して検討が必要な事項を以下に整理する。

| 検討事項 | 内容 |
|----------|------|
| 動画数の下限値 | 「3本以上」を出発点としているが、実際の運用データを踏まえて調整が必要になる可能性がある |
| 更新停止の閾値 | 90日という目安は仮設定。実際の実況シリーズの投稿ペースデータをもとに調整する |
| 完結済みシリーズの扱い | オーナー判断に委ねる方針としているが、完結済みシリーズの登録価値についてポリシーとして明確化する必要がある |
| ショート動画・生配信アーカイブの扱い | 通常の実況動画と区別すべきかどうかの判断基準が未定 |
| 収集候補の品質評価サイクル | 収集・登録後にユーザーの反応（マイリスト登録率）をフィードバックとして判定基準を改善するループの設計 |
