## 1. 調査目的

AI運営者のPhase 2機能（担当ジャンルの再生リストを自律的に発掘・登録する）を実現するにあたり、YouTube Data API v3の利用可能なエンドポイント・クォータコスト・利用規約上の制約を調査し、実現可能性を評価する。

## 2. 利用可能なAPIエンドポイント

### 2.1 コンテンツ収集に関係するエンドポイント一覧

| エンドポイント | 用途 | クォータ/回 |
|--------------|------|:---------:|
| `search.list` | キーワードで動画・再生リスト・チャンネルを検索 | **100** |
| `playlists.list` | 再生リストの詳細情報を取得 | 1 |
| `playlistItems.list` | 再生リスト内の動画一覧を取得 | 1 |
| `channels.list` | チャンネルの詳細情報を取得 | 1 |

### 2.2 search.list の特記事項

`search.list` はコンテンツ探索の起点となる唯一のエンドポイントであるが、コストが突出して高い（1回100ユニット）。

主な検索パラメータは以下の通り。

| パラメータ | 内容 |
|-----------|------|
| `q` | 検索キーワード（例：「ゲーム名 実況」） |
| `type` | 取得するリソース種別。`playlist` を指定することで再生リストのみに絞れる |
| `regionCode` | 地域フィルタ。`JP` を指定することで日本向けコンテンツに絞れる |
| `relevanceLanguage` | 言語フィルタ。`ja` を指定することで日本語コンテンツに絞れる |
| `maxResults` | 1回のリクエストで取得する件数（最大50件） |
| `pageToken` | ページネーション用トークン。次ページ取得に使用 |

> **重要**: `type=playlist` で絞り込んだ場合でも、返却されるのは再生リストの基本情報（ID・タイトル・チャンネルID）のみ。動画数・動画一覧・投稿日時などは別途 `playlistItems.list` または `playlists.list` で取得する必要がある。

### 2.3 playlists.list で取得できる情報

`search.list` でIDを特定した後、`playlists.list` で以下の詳細情報を取得する（コスト1ユニット・最大50件を1リクエストで取得可能）。

| フィールド | 内容 |
|-----------|------|
| `snippet.title` | 再生リストタイトル |
| `snippet.channelId` | チャンネルID |
| `snippet.channelTitle` | チャンネル名 |
| `snippet.publishedAt` | 再生リスト作成日時 |
| `contentDetails.itemCount` | 動画数 |

## 3. クォータコスト試算

### 3.1 1ジャンルあたりの収集コスト（1回の巡回）

AI運営者が担当ジャンルのYouTubeを巡回して再生リスト候補を収集する場合の試算。

| ステップ | API呼び出し | 1回のコスト | 条件 |
|---------|------------|:---------:|------|
| キーワード検索（1クエリ・50件取得） | `search.list` × 1 | 100 | 1キーワードあたり |
| 再生リスト詳細の一括取得 | `playlists.list` × 1 | 1 | 50件を1リクエストで処理 |
| 動画一覧の取得（品質判定用） | `playlistItems.list` × N | N × 1 | 候補件数に比例 |

#### シナリオA：1ジャンル・3キーワード・候補50件で品質チェック

| ステップ | 回数 | コスト |
|---------|:---:|:---:|
| `search.list`（3キーワード） | 3 | 300 |
| `playlists.list`（候補150件を3リクエスト） | 3 | 3 |
| `playlistItems.list`（候補50件に絞って動画取得） | 50 | 50 |
| **合計** | — | **353** |

#### シナリオB：1ジャンル・5キーワード・候補100件で品質チェック

| ステップ | 回数 | コスト |
|---------|:---:|:---:|
| `search.list`（5キーワード） | 5 | 500 |
| `playlists.list`（候補250件を5リクエスト） | 5 | 5 |
| `playlistItems.list`（候補100件に絞って動画取得） | 100 | 100 |
| **合計** | — | **605** |

### 3.2 9ジャンル全体での消費試算

| 条件 | 1回の巡回コスト | 既存Function A/B との合算（上限時） |
|------|:-----------:|:----------------------------:|
| シナリオA × 9ジャンル | 3,177 | 〜5,200（余裕あり） |
| シナリオB × 9ジャンル | 5,445 | 〜7,500（上限に接近） |

> **現状の既存バッチ**: YouTube定期取得仕様書の試算によると、Function A/Bの消費は通常運用で大幅にゼロに近く、最大でも約2,000ユニット/回。デフォルト上限は10,000ユニット/日。

### 3.3 クォータの評価

シナリオAの頻度（例：週1回）であれば、既存バッチとの合算でもデフォルト上限10,000ユニット/日の範囲内に収まる可能性が高い。シナリオBを毎日実行した場合は上限超過のリスクがあるため、実行頻度を週1〜2回に抑えるか、クォータ増枠申請が必要になる。

クォータ増枠はGoogle Cloud Consoleから申請可能。無料で付与されるが、審査においてはYouTube利用規約の遵守を明示する必要がある。

## 4. 利用規約上の制約

### 4.1 スクレイピングの明示的禁止

YouTube API Services Developer Policiesは、APIクライアントによるYouTubeのスクレイピングを明示的に禁止している。

> "You and your API Clients must not, and must not encourage, enable, or require others to, directly or indirectly, scrape YouTube Applications or Google Applications, or obtain scraped YouTube data or content."

また、YouTubeの `robots.txt` は `/channel`・`/playlist`・`/watch` などの主要パスへのアクセスを汎用クローラーに対して禁止している。

**結論**: スクレイピングによるコンテンツ収集は明確に禁止されており、YouTube Data API v3を通じた正規のAPI呼び出しのみが許容される。

### 4.2 APIを通じた自動収集の可否

YouTube Data API v3 を通じた `search.list` による検索・再生リスト情報の取得は、正規のAPI利用として許容される。ただし、以下の点に注意が必要。

| 項目 | 内容 |
|------|------|
| データの保存 | API経由で取得したデータはFirestoreに保存してよい（ゲームタイトル・チャンネル・再生リスト情報の保存は既存機能と同様） |
| 自動化 | Cloud Schedulerによる定期的な自動実行は既存のFunction A/Bと同じ仕組みであり、問題ない |
| 再配布・転載 | 取得したコンテンツ（動画タイトル・サムネイル）をプレミテ上で表示することは既存機能と同様であり、YouTubeの埋め込み・表示ポリシーの範囲内 |
| YouTube Attribution | YouTube APIから取得したデータを表示する際はYouTubeのブランド表示ガイドラインに従う必要がある（既存機能と同様） |

### 4.3 クォータ増枠申請時の注意事項

増枠申請においてGoogleはアプリケーションの利用目的・規約遵守・実績を審査する。スクレイピング・無断転載・規約違反の疑いがある場合は却下される。正当な利用目的（再生リストの発見・登録）を明確に記述し、既存の規約遵守実績を示すことが重要。

## 5. 実現可能性の評価

| 観点 | 評価 | 備考 |
|------|:---:|------|
| 技術的実現可能性 | ○ | `search.list` + `playlists.list` + `playlistItems.list` の組み合わせで収集ロジックは構築可能 |
| クォータ（週1回・9ジャンル） | ○ | デフォルト10,000ユニット/日の範囲内で収まる見込み |
| クォータ（毎日・9ジャンル） | △ | シナリオBでは上限超過リスクあり。増枠申請を検討 |
| 利用規約 | ○ | API経由の正規利用は許容。スクレイピングは禁止（既存方針と同様） |
| 既存スタックとの親和性 | ○ | Cloud Scheduler + Cloud Functions の既存構成をそのまま流用可能 |

**総合評価**: 技術的・規約的にPhase 2の実現は可能。クォータは週1〜2回の実行頻度を基本とし、運用データを見ながら増枠申請の要否を判断する。

## 6. 推奨アーキテクチャ案

### 6.1 概要

既存のFunction A/Bと同様の構成で、新たにFunction Cとして実装する。

```
Cloud Scheduler（週1回、例：日曜 03:00）
    ↓
Function C（AI運営者_コンテンツ収集バッチ）
    ↓
各AI運営者のジャンル情報を取得
    ↓
ジャンルごとに検索キーワードを生成
    ↓
search.list → playlists.list → playlistItems.list
    ↓
品質スクリーニング（Gemini APIで判定）
    ↓
登録候補をFirestoreに保存
    ↓
管理画面でオーナーがレビュー・承認
```

### 6.2 既存機能との分離

Function Cは既存のFunction A/Bとは独立したCloud Functionsとして実装し、クォータ消費の計上もFunction A/Bとは別に管理することが望ましい。

### 6.3 クォータ超過の対策

Function Cの実行前に当日のクォータ消費残量を確認し、残量が不十分な場合は実行をスキップする。スキップ時はFunction A/B同様に管理者に通知する。

## 7. 未解決の検討事項

本調査で回答を得られなかった、または引き続き検討が必要な事項を以下に整理する。

| 検討事項 | 内容 |
|----------|------|
| 検索キーワード設計 | ジャンルごとに有効な検索クエリ（例：「RPG 実況 再生リスト」）の設計が必要。コンテンツ品質基準検討書と合わせて定義する |
| 既存登録済みとの重複排除 | 検索結果に既にプレミテ登録済みの再生リストが含まれる場合の除外ロジックが必要 |
| 収集候補の上限 | 1回の巡回でオーナーに提示する候補件数の上限設計（レビューUIのUXに依存） |
| search.listの精度 | `type=playlist` での検索精度について実際に試験的な呼び出しを行い、不要コンテンツの混入率を検証する必要がある |
| 非公開・削除済み再生リストへの対処 | 検索結果に含まれる再生リストが取得時点で非公開・削除済みの場合のハンドリング |
