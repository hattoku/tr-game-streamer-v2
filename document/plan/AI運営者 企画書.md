## 1. 概要

プレミテはAIを主軸とした運営体制を採用する。その中核を担うのが**AI運営者**である。

AI運営者は、プラットフォーム全体の日常運営を継続的に担う自律型AIエージェントである。人間のアルバイト・ボランティアを前提とした従来型の運営モデルからの意識的な転換として設計されており、AIが主体となってプラットフォームを維持・発展させる体制の根幹をなす。

現時点では**審査フローの一次スクリーニング（STEP 1）**を主な役割として担っているが、将来的にはYouTubeを自律的に巡回し、優良な再生リストを能動的に発掘・登録するコンテンツエージェントへと進化することを目指す。

-----

## 2. 背景と設計思想

### 2.1 なぜ「AI主軸」なのか

プレミテのコンテンツ拡充と品質維持には、以下の業務が継続的に必要となる。

- 投稿された再生リスト登録提案の一次スクリーニング
- スパム・荒らし・通報の対応
- 新規コンテンツの発掘・登録

これらを人間のボランティア・アルバイトに依存する場合、継続性・稼働時間・品質のばらつきが避けられない。一方、AIは審査ルールを与えることで24時間・一定品質で業務を遂行できる。オーナーが少ない工数で最終判断に集中できる体制を実現するために、AI主軸の運営モデルを選択した。

### 2.2 「AI生成コンテンツ氾濫時代」との関係

プレミテの本質的な価値は「人間による本当の評価」の集積にある。AI運営者が担うのはプラットフォームの運営管理であり、レビューや評価はあくまで人間ユーザーが行う。AI運営者は「人間の評価が活きる場を整備するAI」という役割であり、コンテンツの質を判定するのではなく、**良質な評価環境を保つための番人**として機能する。

## 3. AI運営者の定義

### 3.1 システム上の位置づけ

AI運営者は、システム上は**AIフラグが立ったユーザーアカウント**として実装される。ロールは「運営者」と同等であり、すべての審査ワークフローのSTEP 1に自動アサインされる。

### 3.2 命名と配置

| 項目 | 仕様 |
|------|------|
| 命名 | `プレミテ審査AI`（固定） |
| 配置数 | **1体**（プラットフォーム全体で共通） |
| 生成タイミング | システム初期構築時に1体のみ作成 |
| 削除 | 不可（審査履歴の保持のため。無効化で対応） |

> **設計方針**: AI運営者をジャンルごとに分ける設計から、1体に統合する設計に変更した。Gemini APIはファインチューニングをサポートしておらず、ジャンルごとに異なるモデルを持つことはできない。審査時にジャンル・ゲームタイトル情報をプロンプトに動的注入することで、ジャンル専門性と同等の審査品質を実現する。

### 3.3 プロンプトへの情報注入

AI運営者は審査のたびに、対象ワークフローに紐づく以下の情報をプロンプトに含めて判定を行う。

| 注入情報 | 用途 |
|----------|------|
| ゲームジャンル名 | ジャンル固有の審査観点の付与 |
| ゲームタイトル名 | タイトル固有の文脈での判定 |
| 審査対象のコンテンツ詳細 | スパム・品質・完結・更新停止の判定 |

### 3.4 人間の運営者との違い

| 項目 | AI運営者 | 人間の運営者 |
|------|:-------:|:---------:|
| ログイン | 不要（システムが自律実行） | 必要 |
| 稼働時間 | 24時間・自動 | 任意 |
| 削除 | 不可 | 可（ロール剥奪） |
| 有効・無効切り替え | 可 | 可 |
| 担当ジャンル | なし（全ジャンル共通） | 複数選択可 |

## 4. 役割の全体像

AI運営者の役割は、現フェーズと将来フェーズで段階的に拡張される。

### 4.1 現フェーズの役割：審査STEP 1（一次スクリーニング）

ユーザーから提出された以下の審査対象に対し、一次スクリーニングを自動実行する。

| 審査対象 | AI運営者のスクリーニング観点 |
|----------|--------------------------|
| 再生リスト登録提案 | スパム・荒らし検出 / 更新停止の判定 / 完結判定 |
| 再生リスト通報 | スパム・荒らし検出 |
| レビューコメント通報 | スパム・荒らし検出 |
| ゲームタイトル編集提案 | スパム・荒らし検出 |

判定結果は審査ワークフローのSTEP 1コメント欄に自動記録される。問題なしと判定した場合はその根拠のサマリーを、要確認フラグを立てた場合はその理由と根拠を記録し、STEP 2でオーナーが最終判断を下せる状態にする。

審査はAI運営者が自動実行するため、人間の審査者による手動操作は不要である。

### 4.2 将来フェーズの役割：自律的なコンテンツ収集・登録

再生リストを能動的に発掘し、登録候補としてオーナーに提示するエージェントとして機能することを目指す。

収集フローは2段階で構成される。

```
【Stage 1】新作ゲームタイトルの発見（Function D・将来実装）

  楽天ブックスAPI（新着・ランキング）を定期取得
      ↓
  DBに未登録のゲームタイトルを候補として抽出
      ↓
  オーナーに提示 → 承認でゲームタイトルマスタに追加

          ↓ タイトルがDBに蓄積される

【Stage 2】登録済みタイトルの再生リスト収集（Function C・将来実装）

  DBの登録済みゲームタイトル名で YouTube search.list を実行
      ↓
  ハードフィルタ（動画数・重複・公開状態）
      ↓
  Gemini APIによる品質スクリーニング
      ↓
  オーナーに候補提示 → 承認で再生リスト登録
```

> **タイトルベース収集の採用理由**: ジャンル名キーワード（例：`RPG 実況`）による検索は精度が低く、実況者がジャンル名をタイトルに含めないケースも多い。ゲームタイトル名（例：`ドラゴンクエスト11 実況`）で直接検索することで、意図したコンテンツを確実にヒットさせる。

> **楽天ブックスAPIを新作検知に採用する理由**: ゲームタイトルのマスターデータは楽天ブックスAPIで統一する既存設計と一貫性がある。YouTube APIクォータを消費せずに新作情報を取得できる。

### 4.3 将来フェーズで検討中の役割拡張

| 役割 | 概要 |
|------|------|
| STEP 1承認の自律完結 | 将来的にはAI運営者がSTEP 1を完全に自律完結できるよう権限拡張を検討する |
| タグ・ゲームタイトルの自動補完 | コンテンツ収集時に適切なゲームタイトル紐づけやタグの提案も合わせて実施する |
| 品質モニタリング | 登録済み再生リストの更新停止・削除を定期的に検知し、オーナーへの通知やステータス更新を行う |

## 5. フェーズ別ロードマップ

### Phase 1（現在）：審査AIとしての稼働

**目標**: ユーザー提案・通報の一次スクリーニングをAI運営者が自動処理し、オーナーが最終判断業務に集中できる体制を確立する。

- 審査STEP 1の自動実行（全ジャンル共通の1体で対応）
- ジャンル・タイトル情報のプロンプト動的注入による審査品質の確保
- 判定根拠のコメント記録

### Phase 2（中期）：コンテンツ収集エージェントへの進化

**目標**: 楽天ブックスAPIによる新作タイトル検知（Function D）と、ゲームタイトルベースの再生リスト収集（Function C）を実装する。

- Function D：楽天ブックスAPIの新着・ランキングから未登録タイトルを検知
- Function C：DBの登録済みタイトル名でYouTubeを検索し、再生リスト候補を収集
- オーナーへの登録提案ワークフローの整備

### Phase 3（長期）：審査・運営の全自律化

**目標**: オーナーが最終判断に専念できる体制を確立し、日常運営の大半をAI運営者が完結させる。

- STEP 1審査の自律完結（オーナーへのエスカレーションは例外ケースのみ）
- コンテンツの品質維持・モニタリングの自律化

## 6. オーナーとの関係性

AI運営者が自律的に動く範囲が広がるほど、オーナーは例外処理と最終判断に集中できる構造を目指す。

```
【現フェーズ】
  AI運営者 ── STEP 1 スクリーニング ──→ オーナー ── STEP 2 最終承認

  オーナーの関与：全案件の最終承認

【将来フェーズ】
  AI運営者 ── 自律収集・スクリーニング ──→ オーナー ── 例外・最終承認のみ
                  ↓
             通常案件は自律完結
```

AI運営者はオーナーの「代理人」として機能するが、**プラットフォームの方針決定・最終的な掲載可否の責任は常にオーナーが持つ**。これはコンテンツへの責任の所在を明確にするうえで重要な原則である。

## 7. 技術的な位置づけ

AI運営者の機能は現在および将来において、以下の技術スタック上で実装される。

| 機能 | 実装手段 |
|------|----------|
| 審査STEP 1の自動実行 | Firestoreトリガー → Cloud Functions（2nd gen）→ Gemini API |
| 判定結果の記録 | Cloud FunctionsからFirestoreへの書き込み |
| 新作タイトル検知（将来・Function D） | Cloud Scheduler → Cloud Functions → 楽天ブックスAPI |
| 再生リスト収集（将来・Function C） | Cloud Scheduler → Cloud Functions → YouTube Data API v3 |
| 登録候補のオーナー提示（将来） | Firestoreへの登録候補保存 → 管理画面上でのレビューUI |

AIモデルにはGemini API（Flash、最新安定版）を使用する。APIキーはGCP Secret Managerで管理する。

## 8. 課題と検討事項

| カテゴリ | 検討事項 |
|----------|----------|
| プロンプト品質 | ジャンル・タイトル情報の動的注入による審査精度の検証と改善サイクル |
| YouTube API クォータ | Function C（タイトルベース収集）のsearch.list消費はタイトル数に比例するため、登録タイトル数の増加に応じた実行頻度の調整が必要 |
| 新作タイトル検知精度 | 楽天ブックスAPIのランキング・新着データがプレミテの対象プラットフォーム（Switch / Switch 2 / PS5 / PS4）に絞り込めるかの確認 |
| 誤検知・誤登録リスク | 不適切なコンテンツを誤って候補に挙げた場合の対処フロー |
| オーナー承認フロー | 大量候補が提示された場合の効率的なレビューUI設計 |
