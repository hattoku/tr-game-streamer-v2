## 1. 概要

### 1.1 機能概要

通知機能は、ゲムコメ上で発生した各種イベントをログイン済みユーザーに届けるための機能である。

ユーザーにとって価値ある情報（マイリスト登録シリーズの新着動画・運営からのお知らせ・審査結果・お問い合わせ返信）を適切なタイミングで届けることで、ゲムコメへの再訪促進・サービス利用の継続性向上を図る。

### 1.2 通知チャネル

本仕様書が対象とする通知チャネルは以下の2つである。

| チャネル | 概要 |
|---------|------|
| サイト内通知 | ヘッダーのベルアイコン＋通知一覧ページ（`/notifications`）。ゲムコメを開いている時に確認できる |
| FCMプッシュ通知 | ブラウザまたはモバイルアプリへのプッシュ通知。ゲムコメを開いていない時にも届く |

### 1.3 対象外

以下は本仕様書の対象外とする。

| 対象外 | 理由・参照先 |
|--------|------------|
| マイリストカードの🔔NEWバッジ | マイリスト機能仕様書に定義済み |
| インフォメーションバー（お知らせバナー） | 管理_お知らせ管理仕様書・共通UIコンポーネント仕様書に定義済み |
| 管理者向けベルアイコン通知 | 管理_お知らせ管理仕様書 第3章に定義済み |
| お問い合わせ管理者への通知 | 管理_お問い合わせ管理仕様書 第7章に定義済み |
| メール通知（新着動画メール等） | 実装しない |

## 2. 通知種別一覧

ユーザー向け通知の種別を以下に定義する。

| # | 通知種別 | トリガー | サイト内通知 | FCMプッシュ |
|---|---------|---------|:----------:|:---------:|
| 1 | シリーズ新着通知 | マイリスト登録済みシリーズに新着動画が追加された時 | ✅ | ✅ |
| 2 | お知らせ通知 | 管理画面からお知らせが公開された時 | ✅ | ✅ |
| 3 | 審査結果通知 | 自分が提案・通報したワークフローの審査が完了した時 | ✅ | — |
| 4 | お問い合わせ返信通知 | 自分のお問い合わせに管理者から返信が届いた時 | ✅ | ✅ |

### 2.1 各通知種別の詳細

#### シリーズ新着通知

| 項目 | 内容 |
|------|------|
| 発生タイミング | YouTube定期取得（Function A）実行後、マイリスト登録済みシリーズに新着動画が検出された時 |
| 送信対象 | 該当シリーズをマイリスト登録しているすべてのログイン済みユーザー |
| 通知本文 | 「「{再生リストタイトル}」に新しい動画が追加されました」 |
| タップ・クリック遷移先 | 該当再生リストの動画プレーヤーページ（最新話） |

#### お知らせ通知

| 項目 | 内容 |
|------|------|
| 発生タイミング | 管理画面でお知らせが公開（掲載開始日時到達）された時 |
| 送信対象 | 通知許可を付与しているすべてのログイン済みユーザー |
| 通知本文 | お知らせのタイトルをそのまま使用 |
| タップ・クリック遷移先 | トップページ（お知らせのインフォメーションバーが表示される） |

#### 審査結果通知

| 項目 | 内容 |
|------|------|
| 発生タイミング | 自分が提案したワークフロー（再生リスト登録提案・ゲームタイトル登録提案・ゲームタイトル編集提案）または通報が、承認または却下された時 |
| 送信対象 | 該当ワークフローを起票したユーザー本人 |
| 通知本文（承認） | 「{審査種別}が承認されました」 |
| 通知本文（却下） | 「{審査種別}が却下されました」 |
| タップ・クリック遷移先 | 該当コンテンツのページ（承認時）またはトップページ（却下時） |

> **FCMプッシュ通知を対象外とした理由**: 審査完了はリアルタイム性が低く、サイト内通知で十分と判断した。

#### お問い合わせ返信通知

| 項目 | 内容 |
|------|------|
| 発生タイミング | 自分のお問い合わせに管理者がスレッドにメッセージを送信した時 |
| 送信対象 | お問い合わせを送信したユーザー本人（ログイン済みユーザーのみ） |
| 通知本文 | 「お問い合わせに返信が届きました」 |
| タップ・クリック遷移先 | お問い合わせ履歴ページ（`/contact/history`）の該当スレッド |

> **補足**: 未ログインユーザーのお問い合わせはメール折り返しのみ。サイト内通知・プッシュ通知の対象外。

## 4. サイト内通知（ヘッダーベルアイコン）

### 4.1 ベルアイコンの表示仕様

フロントヘッダーの右側（ユーザーアイコンの隣）にベルアイコンを配置する。ログイン済みユーザーのみ表示する。

```
┌──────────────────────────────────────────────────────┐
│  [ロゴ]  ナビゲーション          🔔 ③  [ユーザーアイコン] │
└──────────────────────────────────────────────────────┘
                                 ↑
                         未読件数バッジ（赤丸）
```

| 状態 | 表示 |
|------|------|
| 未読通知あり | ベルアイコン＋未読件数バッジ（赤丸）を表示 |
| 未読通知なし | ベルアイコンのみ表示（バッジなし） |

- 未読件数バッジは99件を上限とし、100件以上の場合は「99+」と表示する。
- ベルアイコンをクリックすると通知ドロップダウンを展開する。

### 4.2 通知ドロップダウン

ベルアイコンクリック時に展開するドロップダウン。

#### レイアウト

```
┌────────────────────────────────────┐
│ 通知                  すべて既読にする │
├────────────────────────────────────┤
│ 🔴 シリーズ新着                      │
│    「〇〇実況」に新しい動画が追加されました│
│    3分前                            │
├────────────────────────────────────┤
│ 🔴 お問い合わせ返信                  │
│    お問い合わせに返信が届きました       │
│    1時間前                          │
├────────────────────────────────────┤
│ 🔴 審査結果                          │
│    再生リスト登録提案が承認されました   │
│    2日前                            │
├────────────────────────────────────┤
│         すべて見る →                │
└────────────────────────────────────┘
```

#### 表示件数

未読通知のみを最大10件表示する。10件を超える場合は「すべて見る」リンクを表示し、通知一覧ページへ遷移する。未読通知が0件の場合は「新しい通知はありません」と表示する。

#### 通知1件あたりの表示要素

| 要素 | 内容 |
|------|------|
| 未読インジケーター | 未読の場合は赤丸（🔴）を表示。既読の場合は非表示 |
| 通知種別 | 「シリーズ新着」「お知らせ」「審査結果」「お問い合わせ返信」 |
| 通知本文 | 通知の概要テキスト。最大2行、超過時は省略記号（…）で切り詰め |
| 経過時間 | 通知生成からの経過時間（例：3分前・1時間前・2日前） |

#### クリック時の動作

| 操作対象 | 動作 |
|---------|------|
| 通知1件をクリック | 個別既読にし、各通知種別の遷移先（第2章参照）へ遷移する |
| 「すべて既読にする」をクリック | ドロップダウン内の全件を既読にする |
| 「すべて見る →」をクリック | 通知一覧ページ（`/notifications`）へ遷移する |
| ドロップダウン外をクリック | ドロップダウンを閉じる |

## 6. FCMプッシュ通知

### 6.1 概要

Firebase Cloud Messaging（FCM）を使用して、ゲムコメを開いていない状態でもユーザーのデバイスに通知を届ける機能。技術実装の詳細はゲムコメ_技術スタック仕様書 第2.9節を参照すること。

### 6.2 対象通知種別

| 通知種別 | FCMプッシュ送信 | 送信対象 |
|---------|:-------------:|---------|
| シリーズ新着通知 | ✅ | 該当シリーズをマイリスト登録し、かつ通知許可を付与しているユーザー |
| お知らせ通知 | ✅ | 通知許可を付与しているすべてのログイン済みユーザー |
| 審査結果通知 | — | — |
| お問い合わせ返信通知 | ✅ | 該当ユーザーが通知許可を付与している場合 |

### 6.3 通知許可取得フロー

```
ユーザーがゲムコメにログイン
    ↓
初回ログイン後または設定ページから通知許可を促す
    ↓
ブラウザの通知許可ダイアログを表示
    ├─ 許可した場合 → FCMトークンを取得しFirestoreに保存
    └─ 拒否した場合 → 通知許可フラグをOFFで保存。再度の自動プロンプトは行わない
```

### 6.4 通知許可の促し方

| タイミング | 表示内容 |
|---------|---------|
| 設定ページのプッシュ通知設定からユーザーが手動で許可を求めた時 | ブラウザの通知許可ダイアログを表示する |

自動的な通知許可プロンプトは表示しない。ユーザーが設定ページから能動的に有効化する方式とする。

### 6.5 FCMトークン管理

技術スタック仕様書 第2.9節の方針に準ずる。

- トークンは端末・ブラウザごとに発行される。1ユーザーが複数デバイスで利用する場合、Firestoreのユーザードキュメントにトークンを配列で保持する。
- FCMから無効トークンエラーが返った場合は、Firestoreから該当トークンを削除する。

### 6.6 プッシュ通知のUI（ブラウザ通知）

| 要素 | 内容 |
|------|------|
| タイトル | 「ゲムコメ」 |
| 本文 | 各通知種別の通知本文（第2章参照） |
| アイコン | ゲムコメのアプリアイコン |
| クリック時 | 各通知種別の遷移先URL（第2章参照）を開く |

### 6.7 送信フロー（シリーズ新着通知の例）

```
YouTube定期取得（Function A）実行
    ↓
新着動画を検出
    ↓
該当シリーズをマイリスト登録しているユーザーを取得
    ↓
各ユーザーのFCMトークンを取得
    ↓
Cloud Functions → FCM Admin SDK → 各デバイスへ送信
    ↓
Firestoreのnotificationsコレクションに通知レコードを生成
    ↓
サイト内通知のベルアイコンに反映
```

## 7. 通知設定

### 7.1 設定の場所

設定ページ（`/settings`）の「通知設定」セクションで管理する。詳細なUI仕様はゲムコメ_設定ページ仕様書に定義する。

### 7.2 設定項目

| 設定項目 | 種別 | デフォルト | 影響範囲 |
|---------|------|:--------:|---------|
| プッシュ通知を受け取る（全体） | ON/OFFトグル | OFF | FCMプッシュ通知の全種別 |
| シリーズ新着をプッシュ通知で受け取る | ON/OFFトグル | ON | シリーズ新着のFCMのみ |
| お知らせをプッシュ通知で受け取る | ON/OFFトグル | ON | お知らせのFCMのみ |
| お問い合わせ返信をプッシュ通知で受け取る | ON/OFFトグル | ON | お問い合わせ返信のFCMのみ |

> **設計思想**: 「プッシュ通知を受け取る（全体）」がOFFの場合、種別ごとの設定に関わらずFCMプッシュ通知はすべて送信しない。種別ごとの設定は全体トグルがONの場合のみ有効となる。

> **サイト内通知はすべてOFF不可**: サイト内ベルアイコン通知はユーザー設定によらず常に有効とする。重要な情報（審査結果・お問い合わせ返信）を確実に届けるための設計。

### 7.3 通知許可とアプリ設定の関係

| ブラウザ許可状態 | アプリ設定（全体） | FCMプッシュ通知 |
|----------------|-----------------|:-------------:|
| 許可済み | ON | 送信する |
| 許可済み | OFF | 送信しない |
| 未許可・拒否 | ON | 送信しない |
| 未許可・拒否 | OFF | 送信しない |

ブラウザの通知許可が取得されていない場合、アプリ設定がONでもプッシュ通知は送信されない。設定ページでは通知許可の状態を表示し、未許可の場合は許可を促す文言を表示する。

## 8. 通知の保持・削除ポリシー

| 項目 | 仕様 |
|------|------|
| 保持期間 | 通知生成日から90日間 |
| 削除方式 | 90日経過した通知をCloud Schedulerが定期的にバッチ削除する |
| ユーザーによる削除 | 初期実装では提供しない |
| 既読状態の保持 | 削除されるまで保持する |
