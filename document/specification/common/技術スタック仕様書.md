## 1. 全体構成

プレミテは Next.js アプリケーションを Google Cloud Run 上で稼働させる構成を採用する。フロントエンドとバックエンド（API Routes）を単一のコンテナにまとめ、別途サーバーを用意しない完全サーバーレス構成である。

```
Next.js（フロント + バックエンド）
    ↓ Dockerコンテナ化
Cloud Run（すべてここで動く）
    ├ フロントエンド（React UI）
    ├ API Routes（バックエンド処理）
    └ ビジネスロジック
    ↓
Firestore（メインDB）
    ├ YouTubeデータ（playlists, videos, channels）
    ├ マスターデータ（games, genres, themes, tags）
    ├ ユーザーデータ（users, reviews, mylist, collections, watch_history, watch_progress）
    └ 運営・システム（workflows, notices, inquiries, operation_logs, batch_logs）

YouTube API取得:
Cloud Scheduler → Function A / Function B（Cloud Functions）→ Firestore

AIキャラクター審査:
Firestore トリガー → Cloud Functions → Gemini API → Firestore

コンテンツ収集（AI運営者 Phase 2）:
Cloud Scheduler → Function C（Cloud Functions）→ YouTube Data API v3 / Gemini API → Firestore

プッシュ通知:
Cloud Functions → Firebase Cloud Messaging → ユーザーデバイス

※ BigQuery は現時点では使用しない。運用が軌道に乗った段階で
　 Firebase Extension による Firestore 自動同期を導入予定。
　 その際、IaC（Terraform）も合わせて整備する。
```

### 1.1 独自ドメインでの動作

```
1. Cloud Run にデプロイ（本番：puremite.net / 検証：stg.puremite.net）
   ↓
2. 独自ドメインを Cloud Run の URL に設定（本番・検証それぞれ）
   ↓
3. ユーザーが puremite.net または stg.puremite.net にアクセス
   ↓
4. Cloud Run 上の Next.js アプリが動作
   ↓
5. 画面表示（すべて Next.js が担当）
```

Next.js だけでフロントエンドもバックエンドも完結する。別途サーバーは不要。

## 2. 技術スタック詳細

### 2.1 アプリケーション

|項目      |技術          |採用理由                                                                  |
|--------|------------|----------------------------------------------------------------------|
|フレームワーク |Next.js 14+ |Cloud Run で最も一般的な構成。フロント + バックエンドを1つのアプリで完結。公式ドキュメント・サンプルが豊富。属人化リスクが低い|
|言語      |TypeScript  |型安全性。Next.js の標準                                                      |
|UI ライブラリ|React       |Next.js の標準                                                           |
|スタイリング  |Tailwind CSS|Next.js との相性が良い。開発速度が速い                                               |
|ホスティング  |Cloud Run   |サーバーレス・自動スケール。コンテナベース。Node.js によりコールドスタートが速い                          |

### 2.2 データベース

|項目        |技術                    |採用理由                                                                                                              |
|----------|----------------------|------------------------------------------------------------------------------------------------------------------|
|メイン DB    |Firestore（Native Mode）|GCP ネイティブ。サーバーレス・スケーラブル。リアルタイム同期。このプロジェクトの要件は NoSQL で実現可能                                                         |
|データモデル    |NoSQL（ドキュメント型）        |スコア計算程度の処理はアプリコードで対応可能。JOIN が不要な設計で Firestore に適合している                                                             |
|分析用 DB（将来）|BigQuery              |Firebase Extension で Firestore から自動同期。SQL による複雑な分析・集計が可能。ベンダーロックイン対策（CSV/JSON エクスポートが可能）。Terraform と合わせて運用軌道後に導入予定|

### 2.3 ゲームデータ取得

|項目      |技術                    |採用理由                                                               |
|--------|----------------------|-------------------------------------------------------------------|
|ゲームカバー画像|楽天ブックス Game Search API|日本市場向けの縦型パッケージ画像が取得可能。日本語ゲームタイトルの検索精度が高い。画像 URL を Firestore に保存して利用|
|ジャンルデータ |楽天ブックス API のカテゴリ情報    |楽天ブックス API に統一することで外部依存を減らす。日本市場に即したカテゴリ体系。ジャンルマスタとして Firestore に保持|

### 2.4 YouTube API 定期取得

詳細は /document/specification/common/共通 youtube定期取得仕様書.md を参照。

Cloud Scheduler → Cloud Functions（第2世代）→ Firestore の構成で定期バッチを実行する。言語は TypeScript（Node.js）で統一する。

### 2.5 AIキャラクター審査 / コンテンツ収集

詳細は /document/specification/management/管理 審査ワークフロー仕様書.md および /document/specification/management/管理_Ai運営者 コンテンツ収集機能仕様書.md を参照。

レビュー審査はワークフロー提出時に Firestore トリガーで Cloud Functions（第2世代）が起動し、Gemini API を呼び出して自動実行する。コンテンツ収集（AI運営者 Phase 2）は Cloud Scheduler により定期起動した Cloud Functions が YouTube Data API v3 と Gemini API を使用して実行する。いずれも判定結果は Firestore に書き込まれる。

|項目        |技術・方針                                                                         |
|----------|-------------------------------------------------------------------------------|
|実行基盤      |Cloud Functions（第2世代）。審査は Firestore トリガー、収集は Cloud Scheduler で起動            |
|AI モデル    |Gemini API を使用する。モデルは随時最新安定版にアップデートする                                        |
|API キー管理  |Secret Manager で管理する。コードへの直書き・環境変数への平文保存は行わない                               |
|言語        |TypeScript（Node.js）。既存の Cloud Functions と統一                                   |

#### Secret Manager の利用方針

Gemini API キーは GCP Secret Manager に登録し、Cloud Functions のサービスアカウントに `roles/secretmanager.secretAccessor` 権限を付与して参照する。キーをコードやリポジトリに含めない。

### 2.6 認証

|項目   |技術                     |採用理由                                                                                                                                                  |
|-----|-----------------------|------------------------------------------------------------------------------------------------------------------------------------------------------|
|認証基盤 |Firebase Authentication|GCP ネイティブ。メール認証・ソーシャルログイン（Google / X）・パスキーに対応。追加コストなし                                                                                                 |
|メール送信|Resend                 |トランザクションメール（本登録確認・パスワード再設定）の要件を満たす。無料枠：月3,000通・1日100通（永続）。MVP 段階では無料で運用可能。Firebase 共有送信ドメインによる迷惑メールリスクを回避できる。Next.js・React との親和性が高く、テンプレートを JSX で記述できる|

### 2.7 ステージング環境の保護（Basic 認証）

ステージング環境（stg.puremite.net）への不正アクセス防止のため、Next.js Middleware による Basic 認証を導入している。

|項目|内容|
|---|---|
|対象環境|ステージング環境（`NEXT_PUBLIC_APP_ENV=stg`）|
|認証方式|HTTP Basic Authentication|
|認証情報管理|GCP Secret Manager に保存し、Cloud Run の環境変数としてマウント|

**設定用環境変数**:
- `STAGING_BASIC_AUTH_USER`: ユーザー名
- `STAGING_BASIC_AUTH_PASSWORD`: パスワード

**メール設定方針**: Firebase Authentication のカスタム SMTP 設定に Resend を接続し、トランザクションメール（本登録確認・パスワード再設定）を Resend 経由で送信する。送信元アドレスはカスタムドメイン（noreply@puremite.net）に設定する。

**トレードオフ**: 外部サービスへの依存が生じる。ただし無料枠（月3,000通）は MVP 段階では十分であり、万が一サービス終了・値上げの際も同等の SMTP サービスへの切り替えコストは低い。

### 2.8 アクセス解析

|項目    |技術                |採用理由                                                       |
|------|------------------|-----------------------------------------------------------|
|タグ管理  |Google Tag Manager|コードを変更せずにタグの追加・変更が可能。Google Analytics との公式連携が容易。無料         |
|アクセス解析|Google Analytics 4|ページビュー・ユーザー行動・流入経路の計測が可能。GTM 経由で導入するため Next.js のコード変更が不要。無料|

### 2.9 PWA 対応

|項目              |技術・方針                                                                                          |
|----------------|-----------------------------------------------------------------------------------------------|
|Web App Manifest|`manifest.json` を配置し、アプリ名・アイコン・テーマカラー（`#E53935`）・表示モード（`standalone`）を定義する。ホーム画面への追加（インストール）を可能にする|
|Service Worker  |`next-pwa` ライブラリを使用して Service Worker を管理する。静的アセットのキャッシュ戦略を設定し、インストール可能条件を満たす                   |
|プッシュ通知基盤       |Firebase Cloud Messaging（FCM）を採用。通知許可取得後に FCM トークンを Firestore に保存する                           |
|通知送信           |Cloud Functions から FCM Admin SDK 経由で通知を送信する                                                   |

#### プッシュ通知の対象

|通知種別       |トリガー                                 |送信対象                    |
|-----------|--------------------------------------|-------------------------|
|シリーズ新着通知  |マイリスト登録済みシリーズに新着動画が追加された時（YouTube定期取得後）|該当シリーズをマイリスト登録しているユーザー   |
|お知らせ通知    |管理画面からお知らせを公開した時                      |通知許可を付与しているすべてのログイン済みユーザー|

#### FCM トークン管理方針

- ユーザーがブラウザで通知許可を付与した時点で FCM トークンを取得し、Firestore のユーザードキュメントに保存する。
- トークンは端末・ブラウザごとに発行される。1ユーザーが複数デバイスで利用する場合、トークンを配列で保持する。
- FCM から無効トークンエラーが返った場合は、Firestore から該当トークンを削除する。

## 3. 将来対応：BigQuery 統合

運用が軌道に乗った段階で、Firestore → BigQuery の自動同期を導入する。現時点では使用しない。

### 3.1 導入方法

Firebase Extension「Export Collections to BigQuery」を使用する。コードを書く必要はなく、設定だけで完結する。

```
Firebase Console
  ↓
Extensions タブ
  ↓
「Export Collections to BigQuery」をインストール
  ↓
設定（数クリック）:
  - エクスポートするコレクション名（例: reviews, playlists）
  - BigQuery データセット名（例: gamecome_production）
  ↓
完了。以降、リアルタイムで増分同期。
```

### 3.2 BigQuery 導入の目的

|目的         |内容                                                                           |
|-----------|-----------------------------------------------------------------------------|
|複雑な分析・集計   |JOIN や集約関数を使った SQL による柔軟な分析が可能になる                                            |
|ベンダーロックイン対策|Firestore のデータを標準的な SQL 形式で保持することで、他システムへの移行時に BigQuery から CSV/JSON エクスポートが可能|
|レポート作成     |月別レビュー数推移・シリーズ別完走率などの KPI を SQL で算出できる                                       |

### 3.3 Terraform との同時整備

BigQuery 導入のタイミングで、GCP リソース全体を Terraform によって IaC 化する。属人化防止と環境再現性の確保が目的。

## 4. 技術選定の根拠

### 4.1 Next.js on Cloud Run

|観点     |内容                                                |
|-------|--------------------------------------------------|
|一般性    |Cloud Run 上で最も広く採用されている構成であり、公式ドキュメントやコミュニティの知見が豊富|
|一体性    |フロントエンドとバックエンドを1つのアプリで完結させることで、別途サーバーの管理が不要       |
|属人化リスク |標準的な構成のため、開発者が変わっても引き継ぎコストが低い                     |
|パフォーマンス|Node.js ベースのため Cloud Run でのコールドスタートが速い            |

### 4.2 Firestore

|観点        |内容                                             |
|----------|-----------------------------------------------|
|GCP との統合  |Cloud Run との親和性が高く、追加の接続設定なしに利用できる             |
|運用コスト     |サーバーレスのため管理作業が不要。スケールも自動対応                     |
|NoSQL の適合性|プレミテのデータモデルは NoSQL で実現可能な範囲に収まっており、複雑な JOIN は不要|
|将来の移行性    |BigQuery 連携により、将来的に他システムへの移行が困難にならない設計         |

### 4.3 楽天ブックス API

|観点    |内容                                          |
|------|--------------------------------------------|
|日本市場適合|日本語ゲームタイトルの検索精度が高く、縦型パッケージ画像が取得できる          |
|統一性   |ゲームデータ取得元を楽天ブックス API に統一することで、外部サービス依存を最小化する|
|コスト   |無料枠での利用が可能                                  |

### 4.4 Firebase Authentication

|観点      |内容                                               |
|--------|-------------------------------------------------|
|機能範囲    |メール認証・Google/X ソーシャルログイン・パスキーをカバーし、プレミテの要件をすべて満たす|
|コスト     |追加費用なし                                           |
|GCP との統合|Firestore・Cloud Run と同一エコシステムで完結する               |

### 4.5 Gemini API（Cloud Functions 経由）

|観点       |内容                                                              |
|---------|----------------------------------------------------------------|
|GCP との統合 |Cloud Functions・Secret Manager・Firestore と同一エコシステムで完結する         |
|既存構成との一貫性|YouTube 定期取得でも Cloud Functions を使用しており、新たなインフラを追加せずに済む          |
|分離性      |AI 処理を Next.js アプリ本体と分離することで、処理遅延や障害が本体に影響しない                  |
|コスト      |Google AI Pro プランに加入済みのため、開発・MVP 段階では追加コストを抑えて利用できる            |

### 4.6 Firebase Cloud Messaging（FCM）

|観点       |内容                                                              |
|---------|----------------------------------------------------------------|
|GCP との統合 |Firestore・Cloud Functions と同一エコシステムで完結する。追加のインフラ構築が不要           |
|既存構成との一貫性|通知送信処理を Cloud Functions に追加するだけで済み、新たなサーバーが不要                  |
|コスト      |無料（送信数に上限なし）                                                    |
|標準準拠     |Web Push 標準（VAPID）に対応しており、iOS Safari・Android Chrome を含む主要ブラウザで動作|
