## 1. 概要

プレミテに登録された再生リスト・動画・チャンネルの情報を、YouTube RSS フィードおよび YouTube Data API v3 を通じて定期的に取得・更新する仕組みを定義する。

処理は更新頻度の異なる2つの独立した Cloud Functions として実装する。頻度の異なる処理を分離することで、不要な API リクエストを排除しクォータ消費を最小化する。

動画更新バッチの新着検知では **RSS フィードを優先し、取得失敗時のみ API にフォールバック**する二段構えの方式を採用する。これにより、通常時の API クォータ消費を大幅に削減する。

-----

## 2. アーキテクチャ

```
Cloud Scheduler
  ├─ 毎日 12:00 / 22:00 ──→ 動画更新バッチ
  └─ 毎週月曜 12:00 ───→ チャンネル更新バッチ

動画更新バッチの取得フロー：
  RSS フィード（優先）
      ↓ 取得失敗時のみ
  YouTube Data API v3（フォールバック）
      ↓
  Firestore
```

| バッチ名 | 実行タイミング | 対象 |
|---|---|---|
| 動画更新バッチ | 毎日 12:00 / 22:00 | 全登録再生リストの動画情報・再生リスト情報 |
| チャンネル更新バッチ | 毎週月曜 12:00 | 全登録チャンネルの基本情報 |

実行タイミングの設計思想：ゲーム実況動画の投稿は夕方〜夜（18時・21時台）に集中する傾向があるため、22:00 のバッチでその日の新着を確実にキャッチする。12:00 のバッチは昼帯に投稿されたものを補足する。

## 3. 動画更新バッチ

### 3.1 実行タイミング

毎日 12:00 および 22:00（Cloud Scheduler による定期実行）

### 3.2 処理概要

Firestore に登録されている全再生リストを対象に、RSS フィードから新着動画を検知する。RSS 取得に失敗した場合は YouTube Data API にフォールバックする。新着が確認された再生リストのみ API で詳細情報を取得し、Firestore を更新する。

```
Firestoreから全再生リストを取得
  ↓
各再生リストに対して：

  ① RSS フィード取得を試みる
      （https://www.youtube.com/feeds/videos.xml?playlist_id={PLAYLIST_ID}）
        ↓ 成功・新着なし → 処理をスキップ（API 呼び出しなし）
        ↓ 成功・新着あり → 新着動画IDを特定 → ③ へ
        ↓ 取得失敗       → ② へ

  ② API フォールバック（YouTube playlistItems.list）
      新着動画IDを特定 → ③ へ

  ③ 新着動画の詳細情報を API で取得
      （タイトル・サムネイル・再生リスト内順番）
        ↓
      Firestoreの動画情報を更新（追加・変更を反映）
        ↓
      YouTube playlists.list で再生リスト情報を取得
        ↓
      Firestoreの再生リスト情報を更新
        ↓
      該当再生リストをマイリスト登録しているユーザーへ新着通知を発行する（詳細は [/document/specification/page/ページ ユーザー通知機能仕様書.md](/document/specification/page/ページ ユーザー通知機能仕様書.md) 参照）。
        ↓
      新着動画が1件以上検知された再生リストに対して：
      プッシュ通知送信処理をトリガー（非同期）
      （詳細は/document/specification/management/管理 FCMプッシュ通知基盤仕様書.md 第6章参照）
```

### 3.3 RSS フィードの仕様

YouTube は再生リスト単位の RSS フィードを公開している。

```
https://www.youtube.com/feeds/videos.xml?playlist_id={PLAYLIST_ID}
```

- 返却形式：Atom（XML）
- 取得件数：最新 15 件
- 取得可能な情報：動画ID・動画タイトル・公開日時・説明文
- YouTube API クォータ：消費なし
- 注意：YouTube の非公式扱いの仕様であり、予告なく変更・廃止される可能性がある。そのため API フォールバックと組み合わせた二段構えで運用する。
- 15件制限について：再生リストの初回登録時に API で全件取得済みであるため、定期更新時は「前回以降の新着差分」だけを検知すればよい。ゲーム実況の投稿ペースは1バッチあたり数本が現実的な想定であり、1バッチで15件を超える新着が発生することはほぼない。

### 3.4 新着判定の方法

RSS フィードから取得した動画IDと公開日時を、Firestore に保存済みの動画リストと比較する。Firestore に存在しない動画ID、または公開日時が更新されているものを新着と判定する。

### 3.5 取得フィールド

#### 動画情報（API で取得）

|フィールド   |API パラメータ                        |用途            |
|--------|---------------------------------|--------------|
|動画ID    |`snippet.resourceId.videoId`     |動画の一意識別子      |
|動画タイトル  |`snippet.title`                  |再生リスト上での表示    |
|動画サムネイル |`snippet.thumbnails`             |再生リスト上での表示    |
|再生リスト内順番|`snippet.position`               |エピソード順の管理     |
|公開日時    |`contentDetails.videoPublishedAt`|新着検知・新着順ソートの基準|

#### 再生リスト情報（API で取得）

|フィールド     |API パラメータ           |用途                                      |
|----------|--------------------|----------------------------------------|
|再生リストタイトル |`snippet.title`     |タイトル変更の反映                               |
|再生リストサムネイル|`snippet.thumbnails`|サムネイル変更の反映                              |
|最新動画公開日時  |動画情報から算出            |`latestVideoPublishedAt` として保存。新着順ソートに使用|

> **補足**：再生リストのサムネイルはチャンネルオーナーが明示的に設定しない場合、再生リスト内の先頭動画のサムネイルが使用される。逆順管理（新着が先頭）のチャンネルでは動画追加のたびにサムネイルが変わるため、定期更新の対象に含める。

### 3.6 クォータ消費の試算

RSS で新着なしと判定された再生リストは API を呼び出さない。クォータ消費は実際の新着数に比例する。

|条件                                     |消費ユニット/回    |
|---------------------------------------|------------|
|登録再生リスト 1,000 件・全件 RSS 成功・新着なし        |0 ユニット      |
|登録再生リスト 1,000 件・全件 RSS 失敗（フォールバック時の上限）|約 2,000 ユニット|
|デフォルトクォータ（10,000 ユニット/日）に対する割合（上限時）  |約 20%（1回あたり）|

通常運用では大半の再生リストで新着がないため、実際のクォータ消費は旧方式（全件 API）と比較して大幅に削減される。

### 3.7 プッシュ通知との連携

Function Aは新着動画を検知した再生リストに対し、プッシュ通知送信処理をトリガーする。
- 通知送信処理はFunction Aの本体処理（Firestoreへの動画情報更新）とは非同期で実行する
- Function Aの成否と通知送信の成否は互いに影響しない
- 新着なしと判定された再生リストに対しては通知処理を実行しない
- 通知送信の詳細仕様は/document/specification/management/管理 FCMプッシュ通知基盤仕様書.mdの第6.1節を参照すること

## 4. チャンネル更新バッチ

### 4.1 実行タイミング

毎週月曜日 12:00（Cloud Scheduler による定期実行）

### 4.2 処理概要

Firestore に登録されている全チャンネルを対象に、YouTube Data API からチャンネルの基本情報を取得し、Firestore を更新する。

```
Firestoreから全チャンネルを取得
  ↓
YouTube channels.list でチャンネル情報を取得
  ↓
Firestoreのチャンネル情報を更新
```

### 4.3 取得フィールド

|フィールド    |API パラメータ                   |用途            |
|---------|----------------------------|--------------|
|チャンネル名   |`snippet.title`             |名前変更の反映       |
|チャンネルアイコン|`snippet.thumbnails`        |アイコン変更の反映     |

> **補足**：チャンネル ID は変化しないため定期更新の対象外とする。初回登録時に取得・保存する。登録者数はコンテンツ収集バッチ（/document/specification/management/管理_AI運営者 コンテンツ収集機能仕様書.md）のスコア計算用途に限定して取得し、このバッチでは取得しない。

## 5. YouTube API 利用方針

- **RSS フィードを優先する**：動画更新バッチの新着検知は RSS フィードで行い、API クォータの消費を最小化する。RSS 取得失敗時のみ API にフォールバックする。
- **再生数・いいね数・コメント数・登録者数は取得しない**：プレミテのスコアリングはプレミテ独自のユーザー行動データで完結する設計であり、YouTube 側の数値には依存しない。頻繁に変動するこれらの数値を保持することは、クォータ消費の増加につながるだけで価値がない。
- **`search.list` は使用しない**：1 リクエスト 100 ユニットと高コストであるため、このバッチでの使用は禁止する（コンテンツ収集バッチは除く）。
- **クォータ超過時の方針**：1 日のクォータ（10,000 ユニット）を超過した場合、その日の残りのバッチ実行はスキップし、翌日のバッチで処理を再開する。同日に複数の Function が実行される場合は 動画更新バッチ を優先する（詳細は第6章参照）。

## 6. クォータ超過時の動作

### 6.1 Function 間の優先順位

クォータが逼迫した状況では、動画更新バッチ（動画情報取得）を チャンネル更新バッチ（チャンネル情報更新）より優先する。

| 優先度 | Function | 理由 |
|:-----:|---------|------|
| 高    | 動画更新バッチ | 新着動画の検知はユーザー体験に直結する。動画が未反映のままでは通知や視聴履歴の精度が低下する |
| 低    | チャンネル更新バッチ | チャンネル名・アイコン・登録者数の更新は週次で十分であり、1週間の遅延はサービス品質に実質的な影響を与えない |

チャンネル更新バッチ が実行される月曜 12:00 の時点で、当日の累計クォータ消費（動画更新バッチ の12:00 実行分）が 8,000 ユニットを超えている場合、チャンネル更新バッチ の実行をスキップする。スキップした場合は翌週の定期実行まで待機する（チャンネル更新バッチ の手動再実行は管理画面から行う）。

> **8,000 ユニットの根拠**：チャンネル更新バッチ のクォータ消費上限は `channels.list` の最大呼び出し回数に依存する。登録チャンネル数が数百件規模であれば消費は数十ユニット程度に収まるが、将来の登録増加を見越して 2,000 ユニット以上の余裕が必要と判断した。運用データを踏まえて閾値を調整する。

### 6.2 実行中にクォータが尽きた場合の動作

動画更新バッチ・チャンネル更新バッチ のいずれも、実行中に累計消費が 10,000 ユニットに達した時点で API 呼び出しを即時停止し、Function 全体を終了する。

```
各再生リスト（またはチャンネル）の処理ループ内で：
  API リクエスト実行
    ↓
  累計消費ユニットを api_quota_logs に記録
    ↓
  累計 ≧ 10,000 ユニット？
    ├─ Yes → ループを中断・Function を終了 → ログに「クォータ超過による中断」を記録
    └─ No  → 次の再生リストへ
```

中断時点で処理済みの再生リストの更新結果は Firestore に保存される。未処理の再生リストは翌日のバッチで自動的に処理される（処理順は固定順のため、次回は先頭から再開する）。

リトライ（同日中の再実行）は行わない。クォータのリセットは翌日 00:00（太平洋時間）であり、次回のバッチ実行（12:00 または 22:00）で自動的に再開される。

### 6.3 管理者への通知

クォータ超過による中断が発生した場合、Cloud Functions は Firestore の `admin_notifications` コレクションに通知レコードを自動生成する。同日中の重複通知は抑制する（当日分のレコードが既に存在する場合は新規生成しない）。

通知の表示仕様は「管理_モニタリング・不正監視仕様書 4章」に準ずる。

### 6.4 クォータ消費の記録

動画更新バッチ・チャンネル更新バッチ の各 API リクエスト実行後、消費ユニット数を Firestore の `api_quota_logs` コレクションに随時記録する。ユニット数は Function 内でリクエスト数から自前で算出する（YouTube Data API はレスポンスにクォータ消費量を含まないため）。

| API エンドポイント       | 消費ユニット |
|----------------------|:---------:|
| `playlistItems.list` | 1         |
| `playlists.list`     | 1         |
| `channels.list`      | 1         |
| `videos.list`        | 1         |
