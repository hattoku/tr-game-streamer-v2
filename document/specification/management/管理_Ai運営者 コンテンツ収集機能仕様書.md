## 1. 概要

本仕様書は、AI運営者が担当ジャンルのYouTubeを定期的に巡回し、プレミテへの登録候補となる再生リストを自律的に収集・スクリーニングするバッチ機能（Phase 2）の実装仕様を定義する。

収集された候補はFirestoreに保存され、管理画面の収集候補レビューUI（管理_AI運営者向け収集候補レビューUI仕様書）を通じてオーナーが確認・承認することで、プレミテへの再生リスト登録が完了する。

コンテンツの収集にはYouTube Data API v3のみを使用する。スクレイピングは利用規約で禁止されているため使用しない（AI運営者_YouTube API調査レポート 第4章参照）。

## 2. アーキテクチャ

```
Cloud Scheduler（週1回 日曜 03:00）
    ↓
コンテンツ収集バッチ
    ↓
① クォータ残量チェック
    ├─ 残量不足 → スキップ通知・終了
    └─ 残量あり → 続行
    ↓
② Firestoreからジャンル一覧・検索キーワードを取得
    ↓
③ ジャンルごとに以下を繰り返す：
    search.list（キーワード検索）
        ↓
    playlists.list（詳細情報取得）
        ↓
    ハードフィルタ（動画数・重複・登録者数下限）
        ↓
    channels.list（登録者数を一括取得）
        ↓
    playlistItems.list（動画一覧取得・フィルタ通過分のみ）
        ↓
    Gemini APIによる品質スクリーニング
        ↓
    スコア計算
    ↓
④ 全ジャンルの候補をスコア順に並べ、上位20件をFirestoreに保存
    ↓
⑤ 実行ログをFirestoreに記録
    ↓
⑥ オーナーに通知
```

### 既存バッチとの関係

コンテンツ収集バッチは既存の動画更新バッチ・チャンネル更新バッチとは独立したCloud Functionsとして実装する。クォータ消費の記録も独立したコレクション（`collection_quota_logs`）で管理し、既存の `api_quota_logs` とは分離する。

## 3. コンテンツ収集バッチ

### 3.1 実行タイミング

| 項目 | 仕様 |
|---|---|
| 実行タイミング | 毎週日曜 03:00（Cloud Scheduler） |
| 対象 | Firestoreに登録されている全ジャンルのAI運営者（有効状態のもの） |

週1回の実行を基本とする。AI運営者_YouTube API調査レポート第3章の試算（シナリオA：9ジャンルで約3,177ユニット/回）に基づき、既存バッチとの合算でデフォルトクォータ10,000ユニット/日の範囲内に収まることを確認している。

### 3.2 処理概要

```
Firestoreから有効なAI運営者一覧を取得
    ↓
AI運営者ごとに以下を順次実行：

  ① 担当ジャンルの検索キーワード一覧を取得
       （Firestore: genre_search_keywords/{genreId}）
       ↓
  ② キーワードごとに search.list を実行
       → 再生リストID一覧を取得（最大50件/キーワード）
       ↓
  ③ 取得した再生リストIDをまとめて playlists.list で詳細取得
       （最大50件を1リクエストで処理）
       ↓
  ④ ハードフィルタを適用
       → 動画数2本以下・登録済みID・取得不可・登録者数1万人未満を除外
       ↓
  ⑤ ジャンルごとにチャンネルIDをまとめて channels.list で登録者数を一括取得
       （最大50件を1リクエストで処理）
       ↓
  ⑥ フィルタ通過分の再生リストに対して playlistItems.list を実行
       → 動画タイトル一覧・公開日時を取得（最大30件）
       ↓
  ⑦ Gemini APIで品質スクリーニング
       → result: "skip"（spam）の場合は除外
       → result: "collect" の場合はスコアを計算して候補リストに追加
       ↓
  ⑧ 全ジャンルの処理が完了したら、候補リスト全体をスコア降順で並べ上位20件を選定

選定後：
  ⑨ 上位20件をFirestoreに収集候補として書き込み
  ⑩ 実行ログを記録
  ⑪ オーナーに収集完了通知を送信
```

> **上位20件の選定タイミング**: スコア計算は全ジャンルの処理完了後にまとめて行い、ジャンルをまたいだグローバルランキングで上位20件を決定する。

### 3.3 ジャンルの処理順序

Firestoreから取得したジャンル一覧を固定順（ジャンルID昇順）で処理する。途中でクォータ残量が不足した場合は、処理済みのジャンルまでの結果を保存して終了し、次回の実行で全件処理を再開する（前回未処理分を優先する機能は初期実装では持たない）。

## 4. APIリクエスト設計

### 4.1 search.list

| パラメータ | 値 |
|---|---|
| `part` | `snippet` |
| `type` | `playlist` |
| `q` | 検索キーワード（第10章参照） |
| `regionCode` | `JP` |
| `relevanceLanguage` | `ja` |
| `maxResults` | `50` |

返却される情報: 再生リストID・タイトル・チャンネルID・チャンネル名・サムネイル

### 4.2 playlists.list

| パラメータ | 値 |
|---|---|
| `part` | `snippet,contentDetails` |
| `id` | 再生リストIDをカンマ区切りで最大50件指定 |

返却される情報: タイトル・チャンネルID・チャンネル名・作成日時・動画数

### 4.3 channels.list

| パラメータ | 値 |
|---|---|
| `part` | `statistics` |
| `id` | チャンネルIDをカンマ区切りで最大50件指定 |

返却される情報: 登録者数（`statistics.subscriberCount`）

ジャンルごとのフィルタ通過再生リストに紐づくチャンネルIDをまとめて1リクエストで取得する（最大50件）。追加クォータコストはジャンルあたり1ユニット、9ジャンル合計で最大9ユニット。

### 4.4 playlistItems.list

| パラメータ | 値 |
|---|---|
| `part` | `snippet,contentDetails` |
| `playlistId` | 対象の再生リストID |
| `maxResults` | `30` |

返却される情報: 動画ID・動画タイトル・再生リスト内順番・動画公開日時

> **30件に制限する理由**: 品質スクリーニングに必要な情報（シリーズらしさ・更新継続性・完結判定）は先頭30件で十分に判定可能であり、クォータコストを抑えるために上限を設ける。

## 5. 収集候補のスコアリングと上限制御

### 5.1 収集上限

1回のバッチで Firestore に保存する収集候補は**全ジャンル合計で上位20件**とする。スコア計算は全ジャンルの処理完了後にまとめて行い、グローバルランキングで上位20件を選定する。

### 5.2 登録者数の下限（初期フェーズ）

登録者数が**1万人未満**のチャンネルはハードフィルタで除外する。

> **初期フェーズ限定の設定**: この下限値はサービス拡大に伴い撤廃・緩和を予定している。将来的には良質なコンテンツであれば登録者数が少なくても評価できる設計に移行する。

### 5.3 スコア計算式

```
スコア = 基礎点(100) − フラグ減点 + 登録者数スコア + 更新継続性スコア
```

#### フラグ減点

| 条件 | 減点 |
|---|---|
| フラグ1個につき | −20点 |
| フラグ4個（最大） | −80点 |

#### 登録者数スコア（ジャンル別係数）

登録者数は対数スコアに変換し、ジャンルの視聴記録ニーズに応じた係数を乗じる。

| 優先度 | ジャンル | 計算式 | 上限 |
|---|---|---|---|
| 高 | RPG・シミュレーション | `log10(登録者数) × 5` | 30点 |
| 中 | 格闘・アクション・シューティング | `log10(登録者数) × 4` | 24点 |
| 低 | スポーツ・レース・テーブル・パズル・教育・学習・その他 | `log10(登録者数) × 3` | 18点 |

> **ジャンル優先度の設計思想**: RPG・シミュレーションはストーリー性が高く視聴記録ニーズに最も合致するため高優先とする。格闘・アクション・シューティングは中程度、スポーツ系・パズル系・教育系は比較的低優先とする。

**計算例（登録者数10万人の場合）:**

| ジャンル | `log10(100,000)` | 係数 | 登録者数スコア |
|---|---|---|---|
| RPG | 5 | × 5 | 25点 |
| 格闘・アクション | 5 | × 4 | 20点 |
| スポーツ・レース | 5 | × 3 | 15点 |

#### 更新継続性スコア

| 最終動画投稿からの経過日数 | スコア |
|---|---|
| 30日以内 | +20点 |
| 31〜90日 | +10点 |
| 91日以上 | +0点 |

### 5.4 スコア計算の実装タイミング

Gemini APIの判定結果（`result: "collect"`）を受け取った後、プログラム側でスコアを計算して候補リストに追加する。スコアはFirestore의 収集候補レコードに `aiScore` フィールドとして保存する。

## 6. Gemini APIによる品質スクリーニング

### 6.1 モデルと呼び出し設定

AI運営者_審査機能仕様書 第5章と同様の設定を使用する。

| 項目 | 仕様 |
|---|---|
| モデル | Gemini Flash（最新安定版） |
| レスポンス形式 | JSON（`response_mime_type: "application/json"` を指定） |
| APIキー管理 | GCP Secret Manager |

### 6.2 システムプロンプト

```
あなたはゲーム実況動画の再生リスト視聴記録プラットフォーム「プレミテ」のコンテンツ収集AIです。
担当ジャンル: {genreName}

YouTubeから取得した再生リスト情報をもとに、プレミテへの登録候補として適切かどうかを判定してください。
以下のJSON形式のみで回答してください。余計なテキスト・マークダウン・説明文は一切含めないでください。

{
  "result": "collect" | "skip",
  "flags": ["flag_key", ...],
  "comment": "判定根拠の説明（日本語・200文字以内）"
}

result の定義:
- "collect" : 収集候補として登録する（フラグがあっても原則 collect）
- "skip"    : 除外する（spam フラグが確実な場合のみ）

flags の定義（該当するものをすべて含める。なければ空配列）:
- "not_gameplay"   : ゲーム実況・プレイ動画のシリーズではないと判断される
- "update_stopped" : 更新が長期間停止していると判断される
- "completed"      : シリーズが完結済みと判断される
- "spam"           : スパム・荒らし・不正コンテンツの疑いが強い
- "non_japanese"   : 日本語コンテンツでないと判断される

判定の方針:
- spam フラグが確実な場合のみ result を "skip" にする
- それ以外のフラグ（not_gameplay・update_stopped・completed・non_japanese）は、
  result を "collect" にしたうえで flags に含めてオーナーの判断に委ねる
- 判断に迷う場合は "collect" を優先する
```

### 6.3 ユーザープロンプト

```
以下の再生リストをプレミテへの登録候補として評価してください。

【再生リストタイトル】
{playlistTitle}

【チャンネル名】
{channelTitle}

【動画数】
{videoCount}本

【動画タイトル一覧（先頭から最大30件）】
{videoTitles（番号付きリスト形式）}

【各動画の公開日時（先頭から最大30件）】
{videoPublishedAts（番号付きリスト形式）}

【最終動画公開日時】
{latestVideoPublishedAt}

【判定実行日時】
{currentDateTime}
```

## 7. Firestoreへの書き込み

### 7.1 収集候補コレクション

収集候補は `collection_candidates` コレクションに保存する。

| フィールド | 型 | 内容 |
|---|---|---|
| `playlistId` | string | YouTubeの再生リストID |
| `playlistTitle` | string | 再生リストタイトル |
| `channelId` | string | チャンネルID |
| `channelTitle` | string | チャンネル名 |
| `thumbnailUrl` | string | 再生リストサムネイルURL |
| `videoCount` | number | 動画数 |
| `latestVideoPublishedAt` | Timestamp | 最終動画公開日時 |
| `subscriberCount` | number | チャンネル登録者数（取得時点） |
| `genreId` | string | 担当ジャンルID |
| `genreName` | string | 担当ジャンル名 |
| `aiOperatorId` | string | 収集したAI運営者のユーザーID |
| `aiResult` | string | `"collect"` |
| `aiFlags` | string[] | フラグキーの配列 |
| `aiComment` | string | AI判定コメント |
| `aiScore` | number | スコア計算結果（第5章参照） |
| `status` | string | `"pending"`（初期値）/ `"approved"` / `"rejected"` / `"registered"` |
| `collectedAt` | Timestamp | 収集日時 |
| `reviewedAt` | Timestamp \| null | オーナーがレビューした日時 |
| `reviewedBy` | string \| null | レビューしたオーナーのユーザーID |
| `reviewComment` | string \| null | オーナーのレビューコメント |

### 7.2 ステータス遷移

```
pending（収集直後）
    ├─ オーナーが承認 → approved
    │       ↓
    │   登録処理実行 → registered（再生リストをプレミテに登録完了）
    └─ オーナーが却下 → rejected
```

### 7.3 重複チェック

収集候補の書き込み前に、同一の `playlistId` がすでに以下のいずれかに存在しないかを確認する。

- `playlists` コレクション（プレミテ登録済み）
- `collection_candidates` コレクションの `status: pending` または `approved`

どちらかに存在する場合はその再生リストをスキップし、収集候補に追加しない。

### 7.4 実行ログ

バッチ完了時に `collection_batch_logs` コレクションに実行ログレコードを書き込む。

| フィールド | 型 | 内容 |
|---|---|---|
| `executedAt` | Timestamp | バッチ実行日時 |
| `processedGenres` | number | 処理したジャンル数 |
| `searchCount` | number | `search.list` の実行回数 |
| `candidatesFound` | number | 収集候補として保存した件数（上位20件） |
| `skippedByFilter` | number | ハードフィルタで除外した件数 |
| `skippedBySpam` | number | スパム判定で除外した件数 |
| `quotaConsumed` | number | 消費クォータユニット数 |
| `status` | string | `"success"` / `"partial"` / `"skipped"` / `"error"` |
| `message` | string \| null | スキップ・エラー時の詳細メッセージ |

## 8. クォータ管理

### 8.1 事前チェック

コンテンツ収集バッチ実行開始時に、当日の `collection_quota_logs` の累計消費量を確認する。推定消費量（ジャンル数 × シナリオA単価：約350ユニット）を加算した場合に3,500ユニットを超える場合はバッチをスキップする。

> **3,500ユニットの根拠**: 既存バッチの最大消費（約2,000ユニット/回×2回=4,000）を考慮し、1日の総消費が10,000ユニットを超えないよう、コンテンツ収集バッチ専用の上限を保守的に設定している。運用データを踏まえて調整する。

### 8.2 クォータ試算（channels.list追加後）

シナリオA（1ジャンル・3キーワード・候補50件）での試算。

| ステップ | API呼び出し | コスト |
|---|---|---|
| `search.list`（3キーワード） | 3回 | 300 |
| `playlists.list`（候補150件を3リクエスト） | 3回 | 3 |
| `channels.list`（チャンネルIDを一括取得） | 1回 | 1 |
| `playlistItems.list`（候補50件） | 50回 | 50 |
| **1ジャンル合計** | — | **354** |
| **9ジャンル合計** | — | **〜3,186** |

`channels.list` の追加によるクォータ増加は9ジャンル合計で**9ユニット**（0.3%未満）であり、クォータへの影響は軽微である。

### 8.3 実行中の消費記録

各APIリクエスト実行後、消費ユニット数を `collection_quota_logs` コレクションに随時記録する。

### 8.4 クォータ超過時の通知

スキップが発生した場合は `admin_notifications` コレクションに通知レコードを作成し、オーナーに通知する（管理_お知らせ管理仕様書 第3章に準ずる）。

## 9. エラーハンドリング

| エラー種別 | 対応 |
|---|---|
| YouTube API呼び出し失敗 | 対象ジャンルの処理をスキップし、次のジャンルへ進む。エラーは実行ログに記録する |
| channels.list呼び出し失敗 | 対象ジャンルの処理をスキップし、次のジャンルへ進む。エラーは実行ログに記録する |
| Gemini API呼び出し失敗 | 最大2回リトライ。失敗した場合は当該再生リストをスキップする（判定なしでの候補追加は行わない） |
| Gemini APIレスポンスのJSONパース失敗 | 当該再生リストをスキップする |
| Firestoreへの書き込み失敗 | リトライなし。エラーを実行ログに記録し、バッチのステータスを `"partial"` にする |

## 10. 検索キーワードの管理

### 10.1 Firestoreでの管理

検索キーワードはジャンルごとに `genre_search_keywords` コレクションで管理する。

| フィールド | 型 | 内容 |
|---|---|---|
| `genreId` | string | ジャンルID |
| `keywords` | string[] | 検索キーワードの配列 |
| `updatedAt` | Timestamp | 最終更新日時 |
| `updatedBy` | string | 更新者のユーザーID |

### 10.2 初期キーワード

AI運営者_コンテンツ品質基準検討書 第5章で定義したジャンルごとのキーワード例を初期値として登録する。

### 10.3 管理画面からの編集

オーナーは管理画面の収集候補レビューUI（管理_AI運営者向け収集候補レビューUI仕様書 第6章）からジャンルごとの検索キーワードを追加・編集・削除できる。
