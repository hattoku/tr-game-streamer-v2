## 1. 概要

本仕様書は、AI運営者がプレミテへの登録候補となる再生リストを自律的に収集するバッチ機能（Phase 2）の設計方針を定義する。

収集フローは2段階で構成される。

- **Function D**：楽天ブックスAPIから新作・話題ゲームタイトルを検知し、オーナーの承認を経てゲームタイトルマスタに追加する
- **Function C**：DBの登録済みゲームタイトル名でYouTubeを検索し、再生リスト候補を収集してオーナーに提示する

収集されたコンテンツの最終的な掲載可否は常にオーナーが判断する。

---

## 2. 全体アーキテクチャ

```
【Function D】新作タイトル検知バッチ
Cloud Scheduler（定期実行）
    ↓
楽天ブックスAPI（新着・売れ筋ランキング）を取得
    ↓
DBに未登録のゲームタイトルを候補として抽出
    ↓
Firestoreに新作候補として保存
    ↓
オーナーに通知 → 管理画面で承認 → ゲームタイトルマスタに追加

          ↓ タイトルがDBに蓄積される

【Function C】再生リスト収集バッチ
Cloud Scheduler（週1回）
    ↓
クォータ残量チェック
    ↓
DBの登録済みゲームタイトル一覧を取得
    ↓
タイトルごとに search.list を実行（`{タイトル名} 実況` 等）
    ↓
playlists.list で詳細取得
    ↓
ハードフィルタ（動画数・重複・公開状態）
    ↓
playlistItems.list（フィルタ通過分のみ）
    ↓
Gemini APIによる品質スクリーニング
    ↓
Firestoreに収集候補として保存
    ↓
オーナーに通知 → 管理画面で承認 → 再生リスト登録
```

### 既存バッチとの関係

Function C・Function Dは既存のFunction A・Function Bとは独立したCloud Functionsとして実装する。クォータ消費の記録も独立したコレクションで管理し、既存の `api_quota_logs` とは分離する。

## 3. Function D：新作タイトル検知バッチ（将来実装）

### 3.1 目的

YouTubeのクォータを消費せずに、発売直後の話題タイトルを検知してゲームタイトルマスタを充実させる。

### 3.2 データソース

楽天ブックスAPIの以下のエンドポイントを利用する。

| エンドポイント | 用途 |
|--------------|------|
| 新着ランキング | 直近の発売タイトルを取得 |
| 売れ筋ランキング | 発売後に話題になっているタイトルを取得 |

対象ジャンル・プラットフォームはプレミテの対象機種（Nintendo Switch / Nintendo Switch 2 / PS5 / PS4）に絞り込む。

### 3.3 処理概要

```
楽天ブックスAPIから新着・ランキングを取得
    ↓
DBのゲームタイトルマスタと突合
    ↓
未登録タイトルのみを候補として抽出
    ↓
Firestoreに new_title_candidates コレクションへ保存
    ↓
オーナーに通知
```

### 3.4 実行タイミング（案）

週1〜2回程度の実行を想定。発売直後のタイトルを取りこぼさないよう、発売日の翌週以内に検知できる頻度を目安とする。詳細は実装時に決定する。

## 4. Function C：再生リスト収集バッチ（将来実装）

### 4.1 収集アプローチ

DBに登録済みのゲームタイトル名を検索キーワードの起点とする**タイトルベース収集**を採用する。

> **採用理由**: ジャンル名キーワード（例：`RPG 実況`）は、実況者がジャンル名をタイトルに含めないケースが多く精度が低い。ゲームタイトル名（例：`ドラゴンクエスト11 実況`）で直接検索することで、意図したコンテンツを確実にヒットさせる。

### 4.2 検索クエリの構成

1タイトルあたり以下のクエリを実行する。

```
{ゲームタイトル名} 実況
{ゲームタイトル名} プレイ
```

> クエリ数はクォータ消費に直結するため、実装時に登録タイトル数との兼ね合いで調整する。

### 4.3 実行タイミング

週1回（Cloud Scheduler）を基本とする。YouTube Data API v3の無料クォータ（10,000ユニット/日）の範囲内に収まるよう、実行頻度はタイトル登録数に応じて調整する。

| search.list の消費 | 計算式 |
|-------------------|--------|
| 1クエリ = 100ユニット | タイトル数 × クエリ数/タイトル × 100 |
| 登録100タイトル・2クエリ/タイトル | 20,000ユニット → 2日分に分割実行が必要 |

> クォータの実態は実装着手時に改めて試算し、実行頻度・分割戦略を確定させること。

### 4.4 ハードフィルタ

APIから取得したデータで機械的に除外する条件。

| 除外条件 | 理由 |
|---------|------|
| 動画数が2本以下 | シリーズとして成立しない |
| 既にDBに登録済みの再生リストID | 重複登録の防止 |
| APIで取得できない（非公開・削除済み） | 視聴不可コンテンツ |

### 4.5 Gemini APIによる品質スクリーニング

ハードフィルタを通過した候補に対し、Gemini APIで以下を判定する。

| 判定観点 | 内容 |
|---------|------|
| 実況シリーズらしさ | 動画タイトル群からゲーム実況のシリーズと判断できるか |
| 更新継続性 | 直近の投稿から90日以内、または平均投稿間隔の3倍以内か |
| スパム・荒らし | 明らかな不正コンテンツでないか |
| 完結済み | 「最終回」等の語句からシリーズ完結を推測できるか |

`spam` 判定のみ自動除外。その他のフラグ（`update_stopped`・`completed`・`not_gameplay`）はオーナー判断に委ねる。

## 5. クォータ管理方針

Function C実行前に当日のYouTube APIクォータ消費残量を確認し、残量が不十分な場合はスキップする。スキップ時はオーナーに通知する。

Function AおよびFunction Bのクォータ消費とは別コレクションで記録・管理する。

---

## 6. 品質スクリーニング設計方針

詳細な品質基準はAI運営者_コンテンツ品質基準検討書を参照。実装着手時に以下の未確定事項を確定させること。

| 検討事項 | 内容 |
|----------|------|
| 1クエリあたりの取得件数 | `maxResults` の設定値（最大50件）とクォータのバランス |
| 品質スクリーニングに渡す動画件数 | `playlistItems.list` の `maxResults` の設定値 |
| 更新停止の閾値 | 90日という目安値の妥当性を運用データで検証 |
| ショート動画・生配信アーカイブの扱い | 通常の実況動画と区別すべきかの判断基準 |

## 8. 未解決の検討事項

| 検討事項 | 内容 |
|----------|------|
| Function Cの分割実行戦略 | 登録タイトル数が増えてクォータ上限に近づいた場合の、複数日にまたがる分割実行ロジックの設計 |
| 新規タイトルのみを対象とした差分収集 | 既収集済みタイトルの再スキャンをスキップしてクォータを節約するロジック |
| Function Dのタイトル候補数の上限 | 1回の検知でオーナーに提示する候補件数の上限設計 |
| search.listの精度検証 | `type=playlist` での検索精度について試験的な呼び出しで不要コンテンツの混入率を検証 |
| 楽天ブックスAPIのプラットフォーム絞り込み精度 | Function Dの対象機種フィルタの実装可否 |
