## 1. 概要

本仕様書は、プレミテ管理画面のシステム設定ページの仕様を定義する。

システム設定ページはオーナー専用のページであり、プラットフォーム全体の動作に影響する設定を一元管理する。

| 設定項目 | 概要 |
|---------|------|
| バッチ処理設定 | 定期実行バッチの一括停止・再開、および動作閾値を管理する |
| AIコンテンツ収集設定 | AIによる収集候補の上限数等を管理する |
| テストモード用会員 | フロントのテストモードウィジェットで使用するテスト専用アカウントを指定する |

## 4. バッチ処理設定

### 4.1 概要

Cloud Schedulerによって定期実行される全バッチ処理（動画更新バッチ・チャンネル更新バッチ・コンテンツ収集バッチ）を一括で停止・再開する機能。

メンテナンス時・バグ調査時・外部API仕様変更への対応時など、バッチによるFirestoreへの書き込みや外部API呼び出しを一時的に止めたい場合に使用する。

**設計方針**: Cloud Schedulerのジョブ自体は停止しない。各Function ofの処理冒頭でFirestoreのフラグを参照し、停止中であれば即時終了する。これにより、Cloud RunやCloud Schedulerの設定を変更することなく、管理画面から即時に停止・再開が完結する。

### 4.2 対象バッチ

| バッチ | 通常の実行タイミング | 停止時の挙動 |
|--------|----------------|-----------|
| 動画更新バッチ（Function A） | 毎日 12:00 / 22:00 | 起動後にフラグを確認し、停止中なら即時終了 |
| チャンネル更新バッチ（Function B） | 毎週月曜 12:00 | 起動後にフラグを確認し、停止中なら即時終了 |
| コンテンツ収集バッチ（Function C） | 毎週日曜 03:00 | 起動後にフラグを確認し、停止中なら即時終了 |

> **対象外**: Firestoreトリガーで起動するCloud Functions（審査AIによるSTEP 1自動実行・不正レビュー検知等）はバッチ処理設定の対象外とする。これらはユーザーアクションに紐づくリアルタイム処理であり、定期バッチとは独立して動作する。

### 4.3 共通クォータ制限

| 設定項目 | 既定値 | 説明 |
|----------|--------|------|
| 1日の最大消費クォータ | 10,000 | YouTube Data APIの1日の上限。この値に達した時点で全バッチが停止する |

### 4.4 Firestoreへの保存

停止状態および閾値は、Firestoreの以下のドキュメントで管理する。

| コレクション | ドキュメントID | フィールド | 型 | 説明 |
|------------|-------------|---------|-----|------|
| `system_settings` | `batch` | `enabled` | boolean | `true`：稼働中 / `false`：停止中（全バッチ共通） |
| `system_settings` | `batch` | `dailyQuotaLimit` | number | 1日の最大クォータ制限 |
| `system_settings` | `batch` | `channelUpdateSkipThreshold` | number | チャンネル更新スキップ閾値 |
| `system_settings` | `batch` | `updatedAt` | timestamp | 最終更新日時 |
| `system_settings` | `batch` | `updatedBy` | string | 最終更新者のUID |

### 4.5 画面構成

```
┌─────────────────────────────────────────────────────────────┐
│ バッチ処理設定                                                │
│                                                             │
│  定期バッチ（動画更新 / チャンネル更新 / コンテンツ収集）の     │
│  一括停止・再開を管理します。                                 │
│                                                             │
│  現在の状態: ● 稼働中                                        │
│  最終変更: 2026-03-23 14:00  by オーナー                     │
│                                                             │
│  [ バッチを停止する ]                                        │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  詳細設定                                                    │
│                                                             │
│  1日の最大消費クォータ  [ 10000 ]  ユニット                   │
│  チャンネル更新スキップ閾値  [ 8000 ]  ユニット               │
│                                                             │
│  [ 保存する ]                                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

停止中の場合は状態表示とボタンが切り替わる。

```
  現在の状態: ⚠ 停止中
  最終変更: 2026-03-23 14:05  by オーナー

  [ バッチを再開する ]
```

### 4.6 操作フロー

#### 停止フロー

```
「バッチを停止する」ボタンをクリック
    ↓
確認ダイアログを表示
    ├─ 「バッチ処理を停止します。定期実行が行われなくなります。よろしいですか？」
    └─ 「停止する」ボタン / 「キャンセル」ボタン
    ↓
Firestoreの system_settings/batch の enabled を false に更新
    ↓
画面の状態表示を「⚠ 停止中」に切り替え
```

#### 再開フロー

```
「バッチを再開する」ボタンをクリック
    ↓
確認ダイアログを表示
    ├─ 「バッチ処理を再開します。次回の定期実行から処理が再開されます。よろしいですか？」
    └─ 「再開する」ボタン / 「キャンセル」ボタン
    ↓
Firestoreの system_settings/batch の enabled を true に更新
    ↓
画面の状態表示を「● 稼働中」に切り替え
```

#### 詳細設定の保存フロー

```
閾値を変更して「保存する」ボタンをクリック
    ↓
バリデーション（数値・範囲チェック）
    ↓
Firestoreの system_settings/batch を更新
    ↓
トースト通知「設定を保存しました」を表示
```

### 4.7 補足事項

- バッチ停止中にCloud Schedulerが定刻でFunctionを起動しても、Functionは即時終了するのみであり、Firestoreへの書き込みや外部API呼び出しは行われない。
- 停止中にスキップされた実行は補完されない。再開後は次回の定期実行タイミングから通常処理が再開される。
- バッチ停止はCloud Schedulerのジョブ設定やCloud Functionsのデプロイには影響しない。

## 5. AIコンテンツ収集設定

### 5.1 概要

AI運営者によるコンテンツ収集バッチ（Function C）の動作パラメータを設定する。詳細は「管理_Ai運営者 コンテンツ収集機能仕様書」を参照。

### 5.2 設定項目

| 設定項目 | 既定値 | 説明 |
|----------|--------|------|
| 収集候補保存上限数 | 20 | 1回の実行で `collection_candidates` に保存する上位件数 |
| 実行開始クォータ残量閾値 | 3,500 | 推定消費量を加算してこの値を超える場合、実行をスキップする |

### 5.3 Firestoreへの保存

| コレクション | ドキュメントID | フィールド | 型 | 説明 |
|------------|-------------|---------|-----|------|
| `system_settings` | `ai_collection` | `rankingLimit` | number | 収集候補保存上限数 |
| `system_settings` | `ai_collection` | `quotaThreshold` | number | 実行開始クォータ残量閾値 |

### 5.4 画面構成

```
┌─────────────────────────────────────────────────────────────┐
│ AIコンテンツ収集設定                                          │
│                                                             │
│  収集候補保存上限数  [ 20 ]  件                               │
│  実行開始クォータ残量閾値  [ 3500 ]  ユニット                 │
│                                                             │
│  [ 保存する ]                                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 6. テストモード用会員

### 6.1 概要

フロントのテストモードウィジェット（共通 uiコンポーネント フロント 仕様書 第7章）において、ログイン状態に切り替える際に使用するテスト専用のアカウントを指定する。

テストモード用会員の定義・権限・制約は「管理_ユーザー管理仕様書 第5章」を参照。本節ではシステム設定ページ上での設定操作のみを定義する。

### 6.2 画面構成

#### 未指定の場合

```
┌─────────────────────────────────────────────────────────────┐
│ テストモード用会員                                            │
│                                                             │
│  現在の指定: 未指定                                          │
│                                                             │
│  [ 指定する ]                                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 指定済みの場合

```
┌─────────────────────────────────────────────────────────────┐
│ テストモード用会員                                            │
│                                                             │
│  現在の指定: テストユーザー（test@example.com）               │
│                                                             │
│  [ 変更する ]  [ 指定を解除する ]                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.3 指定フロー

```
「指定する」または「変更する」ボタンをクリック
    ↓
ユーザー検索モーダルを表示
    ├─ メールアドレスまたはユーザー名で検索
    └─ 対象ユーザーを選択して「指定する」ボタンをクリック
         ↓
      確認ダイアログを表示
         ├─ 「{ユーザー名}（{メールアドレス}）をテストモード用会員に指定しますか？」
         └─ 「指定する」ボタン / 「キャンセル」ボタン
              ↓
           Firestoreに保存・画面に反映
```

### 6.4 解除フロー

```
「指定を解除する」ボタンをクリック
    ↓
確認ダイアログを表示
    ├─ 「テストモード用会員の指定を解除しますか？」
    └─ 「解除する」ボタン / 「キャンセル」ボタン
    ↓
Firestoreから指定を削除・画面に反映
    ↓
テストモードウィジェットのログイン切り替えボタンが非活性になる
```

## 7. 操作ログ

システム設定ページで行われたすべての変更操作は操作ログとして記録する。操作ログの閲覧仕様は管理_操作ログ仕様書を参照。

| 操作 | 記録内容 |
|------|---------|
| バッチ処理の停止 | 操作日時・操作者・変更後の状態（停止） |
| バッチ処理の再開 | 操作日時・操作者・変更後の状態（稼働中） |
| バッチ詳細設定の変更 | 操作日時・操作者・変更した項目名・変更前後の値 |
| AI収集設定の変更 | 操作日時・操作者・変更した項目名・変更前後の値 |
| テストモード用会員の指定 | 操作日時・操作者・指定したユーザーID |
| テストモード用会員の変更 | 操作日時・操作者・変更前ユーザーID・変更後ユーザーID |
| テストモード用会員の解除 | 操作日時・操作者・解除したユーザーID |
