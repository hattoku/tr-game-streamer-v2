## 1. 概要

本仕様書は、AI運営者が審査ワークフローのSTEP 1（一次スクリーニング）を自動実行する機能の実装仕様を定義する。

一般ユーザーから提案・通報が提出されると、Firestoreへの書き込みをトリガーにCloud Functionsが起動し、Gemini APIを呼び出してAI審査を実行する。判定結果はSTEP 1の審査履歴コメントとしてFirestoreに書き込まれ、管理_審査ワークフロー仕様書に定める通常の審査フローと接続される。

AI審査が扱う審査種別・判定観点・判定結果の定義については管理_審査ワークフロー仕様書 第5章を参照すること。本仕様書はその実装詳細（トリガー・入力データ・プロンプト・出力形式・DB書き込み）を規定する。

---

## 2. 処理の全体フロー

```
一般ユーザーが提案・通報を提出
    ↓
Firestoreの workflows コレクションにレコード作成
    ↓
Firestoreトリガー → Cloud Functions 起動
    ↓
STEP 1 スキップ条件の確認
    ├─ スキップ対象 → 処理を終了（AI審査は実行しない）
    └─ 対象あり → 続行
    ↓
Firestoreから審査に必要なデータを取得
    ↓
Gemini API に審査リクエストを送信
（ジャンル名・ゲームタイトル名をプロンプトに動的注入）
    ↓
判定結果（JSON）を受信・パース
    ↓
Firestoreのワークフローレコードに判定結果を書き込み
    ↓
ステータスを「審査中 Lv2」に更新
（「問題なし」「要確認フラグあり」いずれの場合も Lv2 へ移行する）
```

> **STEP 1の通過ルールについて**: 管理_審査ワークフロー仕様書 第4章の通り、AI運営者を含む審査者のうち誰か1名が承認すれば通過となる。AI運営者が「問題なし」と判定した場合はSTEP 1を通過（審査中 Lv2 へ移行）する。「要確認フラグあり」の場合も、AI運営者は審査を完了した扱いとしてSTEP 1を通過させ、フラグの内容をコメントに残した上でSTEP 2へ引き継ぐ。

## 3. トリガーと実行条件

### 3.1 トリガー

| 項目 | 仕様 |
|------|------|
| トリガー種別 | Firestoreドキュメント作成トリガー |
| 対象コレクション | `workflows` |
| 対象イベント | `onCreate` |

### 3.2 STEP 1 スキップ条件

以下のいずれかに該当する場合、Cloud FunctionsはAI審査を実行せずに終了する。管理_審査ワークフロー仕様書 第4章のスキップルールと対応する。

| 条件 | 理由 |
|------|------|
| `workflowType` が `game_title_register` | ゲームタイトル登録提案は常にSTEP 1スキップ |
| `genreId` が null または未設定 | ジャンル不明のためSTEP 1スキップ |

### 3.3 AI運営者の取得

Firestoreの `ai_operators` コレクションから、有効状態の唯一のAI運営者レコードを取得する。AI運営者が存在しない（無効化されている等）場合はAI審査を実行しない。

> **設計変更点**: 旧設計では `genreId` をキーにジャンル担当AIを特定していたが、AI運営者が1体に統合されたため、ジャンル検索は不要。有効なAI運営者レコードを1件取得するだけでよい。

### 3.4 重複実行防止（冪等性）

Cloud Functions の at-least-once 保証により、同一のワークフローに対してトリガーが複数回発火する可能性がある。重複実行による二重課金やデータ不整合を防ぐため、以下のチェックを行う。

| 項目 | 仕様 |
|------|------|
| 判定基準 | `workflows/{workflowId}.step1History` 配列内に `isAI: true` の要素が既に存在するか |
| スキップ時の挙動 | 後続の処理（Gemini API 呼び出し・DB書き込み・通知作成）を一切行わず、正常終了する |
| ログ出力 | 「Duplicate execution detected. WorkflowId: {id}」等のログを出力する |

> **注意**: すでに一度 AI 審査が完了している場合、その結果は正として扱い、2回目以降のトリガーは無視する。追加の通知やエラー記録は行わない。

## 4. 審査種別ごとの入力データ

Cloud Functionsは以下のデータをFirestoreから取得し、Gemini APIへの入力として使用する。

### 4.1 再生リスト登録提案（`playlist_register`）

| データ項目 | 取得元 | 用途 |
|-----------|--------|------|
| 再生リストタイトル | `workflows.proposalData.playlistTitle` | スパム・内容判定 |
| チャンネル名 | `workflows.proposalData.channelTitle` | スパム判定 |
| 動画数 | `workflows.proposalData.videoCount` | 更新停止・完結判定の補助 |
| 動画タイトル一覧（最大50件） | `workflows.proposalData.videoTitles` | 完結判定 |
| 各動画の公開日時一覧 | `workflows.proposalData.videoPublishedAts` | 更新停止判定 |
| 最終動画公開日時 | `workflows.proposalData.latestVideoPublishedAt` | 更新停止判定 |
| 審査実行日時 | Cloud Functions実行時の現在時刻 | 更新停止判定（経過期間算出） |
| ゲームジャンル名 | `genres/{genreId}.name` | プロンプト動的注入 |
| ゲームタイトル名 | `game_titles/{gameTitleId}.title` | プロンプト動的注入 |

### 4.2 再生リスト通報（`playlist_report`）

| データ項目 | 取得元 | 用途 |
|-----------|--------|------|
| 再生リストタイトル | `workflows.reportData.playlistTitle` | スパム・荒らし判定 |
| チャンネル名 | `workflows.reportData.channelTitle` | スパム・荒らし判定 |
| 通報理由 | `workflows.reportData.reason` | スパム・荒らし判定 |
| ゲームジャンル名 | `genres/{genreId}.name` | プロンプト動的注入 |
| ゲームタイトル名 | `game_titles/{gameTitleId}.title` | プロンプト動的注入 |

### 4.3 レビューコメント通報（`review_report`）

| データ項目 | 取得元 | 用途 |
|-----------|--------|------|
| 通報対象コメント本文 | `workflows.reportData.commentBody` | スパム・荒らし判定 |
| 通報理由 | `workflows.reportData.reason` | スパム・荒らし判定 |
| ゲームジャンル名 | `genres/{genreId}.name` | プロンプト動的注入 |
| ゲームタイトル名 | `game_titles/{gameTitleId}.title` | プロンプト動的注入 |

### 4.4 ゲームタイトル編集提案（`game_title_edit`）

| データ項目 | 取得元 | 用途 |
|-----------|--------|------|
| 変更前ゲームタイトル情報 | `workflows.proposalData.before` | スパム・荒らし判定 |
| 変更後ゲームタイトル情報（提案内容） | `workflows.proposalData.after` | スパム・荒らし判定 |

| ゲームタイトル名 | `game_titles/{gameTitleId}.title` | プロンプト動的注入 |

## 5. Gemini APIへの入力設計

### 5.1 モデルと呼び出し設定

| 項目 | 仕様 |
|------|------|
| モデル | Gemini Flash（最新安定版） |
| 呼び出し方法 | Cloud FunctionsからGemini API（REST）を呼び出す |
| APIキー管理 | GCP Secret Manager |
| レスポンス形式 | JSON（`response_mime_type: "application/json"` を指定） |

### 5.2 システムプロンプト

すべての審査種別に共通して以下のシステムプロンプトを使用する。`{genreName}` および `{gameTitleName}` は審査時にFirestoreから取得した値を動的に注入する。

```
あなたはゲーム実況動画の再生リスト視聴記録プラットフォーム「プレミテ」の審査AIです。

対象ゲームジャンル: {genreName}
対象ゲームタイトル: {gameTitleName}

ユーザーから提出された提案・通報を審査し、以下のJSON形式のみで回答してください。
余計なテキスト・マークダウン・説明文は一切含めないでください。

{
  "result": "ok" | "flag",
  "flags": ["flag_key", ...],
  "comment": "判定根拠の説明（日本語・200文字以内）"
}

result の定義:
- "ok"   : 問題なし。STEP 2へ引き継ぐ。
- "flag" : 要確認フラグあり。STEP 2でオーナーが判断する。

flags の定義（該当するものをすべて含める。okの場合は空配列）:
- "spam"          : スパム・荒らし・明らかに不正な内容が検出された
- "update_stopped": 動画の更新が長期間停止していると判断される
- "completed"     : 再生リストのシリーズが完結していると推測される

comment の定義:
- 判定の根拠を簡潔に記述する。
- "ok" の場合も根拠サマリーを記述する。
- "flag" の場合はフラグごとの理由を記述する。
```

### 5.3 審査種別ごとのユーザープロンプト

#### 再生リスト登録提案

```
以下の再生リスト登録提案を審査してください。

【再生リストタイトル】
{playlistTitle}

【チャンネル名】
{channelTitle}

【動画数】
{videoCount}本

【各動画のタイトル（先頭から最大50件）】
{videoTitles（番号付きリスト形式）}

【各動画の公開日時（先頭から最大50件）】
{videoPublishedAts（番号付きリスト形式）}

【最終動画公開日時】
{latestVideoPublishedAt}

【審査実行日時】
{currentDateTime}

審査観点:
1. スパム・荒らし: 再生リストタイトル・チャンネル名に明らかに不正・無関係な内容がないか
2. 更新停止の判定: 動画の平均投稿間隔と最終投稿からの経過期間を比較し、明らかな放置かどうかを判断する
3. 完結判定: 動画タイトルからシリーズの完結を推測できるかを評価する（「最終回」「クリア」「エンディング」等の語句を参考にする）
```

#### 再生リスト通報

```
以下の再生リスト通報を審査してください。

【通報対象の再生リストタイトル】
{playlistTitle}

【チャンネル名】
{channelTitle}

【通報理由】
{reason}

審査観点:
1. スパム・荒らし: 通報内容が明らかに嫌がらせ・虚偽・無関係な内容でないかを判断する
```

#### レビューコメント通報

```
以下のレビューコメント通報を審査してください。

【通報対象のコメント本文】
{commentBody}

【通報理由】
{reason}

審査観点:
1. スパム・荒らし: コメント内容および通報理由から、明らかな不正・嫌がらせ・虚偽通報でないかを判断する
```

#### ゲームタイトル編集提案

```
以下のゲームタイトル編集提案を審査してください。

【変更前】
{before（項目名: 値 の形式）}

【変更後（提案内容）】
{after（項目名: 値 の形式）}

審査観点:
1. スパム・荒らし: 変更内容が明らかに不正・無関係・荒らし目的でないかを判断する
```

## 6. 判定結果の出力形式

Gemini APIからのレスポンスは以下のJSON形式で受け取る。Cloud FunctionsはこのJSONをパースし、Firestoreへの書き込みに使用する。

```json
{
  "result": "ok",
  "flags": [],
  "comment": "スパム・荒らし該当なし。更新頻度・完結状況に問題なし。直近の投稿から3日経過しているが平均間隔（4日）の範囲内。"
}
```

```json
{
  "result": "flag",
  "flags": ["update_stopped", "completed"],
  "comment": "最終投稿から180日経過しており、平均投稿間隔（7日）と大きく乖離している。また動画タイトルに「最終回」が含まれており完結の可能性がある。"
}
```

### JSONパースの失敗時

Gemini APIのレスポンスが有効なJSONでない場合、Cloud FunctionsはこれをエラーとしてFirestoreに記録し、AI審査を失敗扱いとする（第8章参照）。

## 7. Firestoreへの書き込み

### 7.1 審査履歴の追記

AI審査完了後、`workflows/{workflowId}` ドキュメントの `step1History` 配列に以下のレコードを追加する。

| フィールド | 型 | 内容 |
|-----------|-----|------|
| `executedAt` | Timestamp | AI審査実行日時 |
| `operatorId` | string | AI運営者のユーザーID |
| `operatorName` | string | AI運営者の表示名（例：プレミテ審査AI） |
| `isAI` | boolean | `true` |
| `action` | string | `"approved"` 固定（AI運営者は常に審査を完了させる） |
| `result` | string | `"ok"` または `"flag"` |
| `flags` | string[] | フラグキーの配列（なければ空配列） |
| `comment` | string | 判定コメント |

> **書き込み時の注意（冪等性の担保）**: 書き込み前および書き込み処理（トランザクション等）の中で、再度 `step1History` に AI 審査結果が存在しないかを確認すること。万が一、API 呼び出しと書き込みの間に別のインスタンスが書き込みを完了していた場合、二重書き込みを回避して処理を終了する。

### 7.2 ワークフローステータスの更新

AI審査完了後、`workflows/{workflowId}` ドキュメントの `status` フィールドを更新する。

| AI判定結果 | 更新後のステータス | 理由 |
|-----------|-----------------|------|
| `"ok"` | `"reviewing_lv2"` | STEP 1通過、STEP 2へ移行 |
| `"flag"` | `"reviewing_lv2"` | AI運営者はフラグ付きでもSTEP 1を完了扱いにし、STEP 2でオーナーが最終判断する |

> AI運営者は「問題なし」「要確認フラグ」いずれの場合もSTEP 1を完了させる。フラグがある場合は判定コメントに理由を記録することで、STEP 2のオーナー判断を補助する。

### 7.3 管理者への通知レコード作成

ステータスを `"reviewing_lv2"` に更新した後、`admin_notifications` コレクションに通知レコードを作成する。詳細は管理_お知らせ管理仕様書 第3章を参照。

## 8. エラーハンドリング

### 8.1 エラー種別と対応

| エラー種別 | 対応 |
|-----------|------|
| Gemini APIの呼び出し失敗（タイムアウト・ネットワークエラー） | 最大2回リトライ。3回失敗でエラーレコードをFirestoreに書き込み、処理を中断 |
| Gemini APIのレスポンスが有効なJSONでない | リトライなし。エラーレコードをFirestoreに書き込み、処理を中断 |
| AI運営者が存在しない・無効化されている | AI審査をスキップ。ワークフローはSTEP 1待機のまま（人間の審査者がいれば人間が審査する） |
| 必要なデータがFirestoreに存在しない | エラーレコードをFirestoreに書き込み、処理を中断 |

### 8.2 エラー時のワークフロー状態

AI審査がエラーで中断した場合、ワークフローのステータスは `"reviewing_lv1"` のまま維持する。ステータスを自動更新しないことで、人間の審査者（オーナーまたは担当運営者）が手動でSTEP 1を処理できる状態を保つ。

### 8.3 エラーレコードの構造

`workflows/{workflowId}` ドキュメントの `step1Errors` 配列に以下のレコードを追加する。

| フィールド | 型 | 内容 |
|-----------|-----|------|
| `occurredAt` | Timestamp | エラー発生日時 |
| `errorType` | string | エラー種別（`api_error` / `parse_error` / `data_missing`） |
| `message` | string | エラーの詳細メッセージ |

### 8.4 管理者へのエラー通知

AI審査がエラーで中断した場合、速やかに `admin_notifications` コレクションに通知レコードを作成し、管理者に手動対応を促す。

| フィールド | 内容 |
|-----------|------|
| `type` | `"ai_review_error"` |
| `title` | `【要対応】AI審査エラー（{審査種別}）` |
| `body` | `AI審査中にエラーが発生しました。詳細はワークフロー詳細画面を確認し、手動での審査をお願いします（エラー内容：{詳細メッセージ}）` |
| `relatedId` | `workflows/{workflowId}` のドキュメントID |
| `targetRole` | `"all_admins"` |
